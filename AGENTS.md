# 项目介绍

这个仓库是在 xv6-riscv 教学操作系统上做的本科毕设增强版，核心目标是把 xv6 扩展成一个可以支撑“实现、验证、实验、展示”完整闭环的研究平台。

项目主线分成四块：

1. 模块一：调度算法优化和内存管理优化(虽然实现了伙伴算法的物理内存分配器，但是并不准备写入论文)。
2. 模块二：统一测试与回归验证。
3. 模块三：实验采集、校验和分析。
4. 模块四：Web 展示和运行态监控。

## 文档组织

正式文档只保留四类入口：

1. `docs/review/`：审查报告，只放审查类材料。
2. `docs/bug/`：bug 档案，只记录开发和排障过程中遇到的问题。
3. `docs/module1-4/`：模块文档，按模块分别组织。
4. `docs/开发进度.md`：开发进度快照。

其他临时稿、旧草稿和迁移中内容都不应作为正式入口。

## 工作约束

1. 每完成一个小功能或一个小修复，就直接提交 `commit`，提交信息要写详细，说明改了什么、为什么改、怎么验证。
2. 每次代码改动后，必须同步更新相关文档，不要只改代码不改文档。
3. 如果修的是 bug，要把处理过程写进 `docs/bug/`，严重 bug 要按 `BUG-014` 这种完整排障报告的风格写。
4. 文档要保持收口，不要为了临时说明再新建一堆无关章节或散页。
5. 在写代码时，多想一想有没有更好的架构设计和实现方式，不要只满足于能跑。
6. 尽量保证工作流程健康，每一次大的改动都能被 `git` 和文档追溯到，方便日后复盘和回滚。
7. 对齐文档时尽量采用追加新节、新子文档或新索引的方式，只有当原文本身记录了错误时才覆盖修正。
8. 不要回滚你没有改过的内容；如果工作区里有别人的改动，先避开它们再继续。

## 写作和提交习惯

- 代码和文档尽量成对更新。
- bug 记录至少要包含：现象、触发条件、排查过程、根因、修复、验证。
- 提交信息优先写成“范围 + 动作 + 结果”的形式，保持详细但直接。

## Git 分支与提交约束

### 分支角色

1. `baseline`：只保留对照组、基线实验和与 baseline 可比性直接相关的改动，不承载新功能主线开发。
2. `scheduler`：模块一调度主线分支，只放调度框架、调度算法和相关最小配套改动。
3. `memory`：模块一内存主线分支，只放 `COW/Lazy/mmap/Buddy` 和地址空间语义相关改动。
4. `testing`：集成验证分支，承接模块二、模块三、模块四，以及跨模块联调后的收口提交。
5. 临时开发分支统一从对应主线分支切出，命名建议为 `topic/moduleX-xxx`、`fix/moduleX-xxx`，合入后尽快删除，不长期堆积。

### 提交粒度

1. 一个提交只解决一个清晰的问题，或者完成一组强相关的小改动，不能把“代码修复 + 文档搬家 + UI 微调 + 数据目录整理”混在一起。
2. 行为变化和纯重构尽量分开提交；代码修复和文档对齐可以成对出现，但不要把多个无关修复塞进同一次提交。
3. 生成物、运行日志、临时数据、截图、草稿默认不进正式历史；只有当它们是长期维护所需的正式样例或基准产物时才纳入跟踪。
4. 模块一、二、三、四跨模块改动只有在确实属于“联调闭环”时才能出现在同一个提交里，否则应先在各自主线分开提交，再到 `testing` 收口。

### 提交信息规范

1. 标题优先使用“范围: 动作 + 结果”的形式，例如：
   - `module1/sched: fix policy switch lock ordering`
   - `module4/dashboard: split router views and unify monitor entry`
   - `docs/review: append module4 re-review with live validation`
2. 范围建议尽量稳定，优先从下面选：
   - `module1/sched`
   - `module1/memory`
   - `module2/tests`
   - `module3/experiments`
   - `module4/dashboard`
   - `docs`
   - `repo`
3. 禁止使用过于模糊、无法复盘的标题，例如：
   - `fix`
   - `update`
   - `优化模块三`
   - `加入 runtime`
   - `模块四的雏形，待修正的地方很多`
   - `阶段7`、`阶段12` 这类只有过程编号、没有结果语义的写法
4. 正式提交尽量补正文，至少写清楚三件事：
   - 改了什么
   - 为什么要改
   - 怎么验证

### 历史整理策略

1. 未发布、未共享的本地临时分支可以整理历史；已经共享或已经作为他人协作基础的分支，不要随意强推重写。
2. 要整形历史时，优先新建一个干净分支，按主题 `cherry-pick` 或整理后重放，不要在脏工作区里直接大规模改写现有主线。
3. 对已经明显成型的一串提交，只允许在“发布前最后收口”时做有限 squash；不要把已经可复盘的稳定历史反复揉碎。
4. 小修补提交如果只是给上一提交擦尾巴，应尽量在本地收口后再发布，避免主线上连续出现很多“修修补补”的碎提交。
5. 如果某段历史已经公开但质量一般，优先追加“对齐文档 + 后续清理提交”修正，而不是为了好看去冒险重写公共历史。

### 当前仓库的历史坏味道示例

1. `scheduler`、`memory` 分支里早期大量“阶段X”式提交，过程可读，但结果语义不够清楚，后续应逐步按“基础设施 / 算法 / 系统调用 / 测试 / 文档”重组。
2. `testing` 分支里存在少量标题过于模糊的提交，例如 `fix`、`优化模块三`、`加入 runtime`，这类写法后续不要再出现。
3. 模块四早期有“雏形”“待修正很多”这种提交标题，说明当时提交的是阶段感受，不是稳定结果；今后应改成“问题 + 动作 + 结果”的可复盘表述。
4. 文档收口之后，`docs` 相关提交已经开始变规范；后续应把这种写法推广到代码分支，不要再回退到“阶段式流水账”。

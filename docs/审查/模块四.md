# 模块四审查记录

## 结论

当前模块四的总体方向是合理的：

- xv6 侧负责采样与导出
- 宿主机 bridge 负责聚合与转译
- 浏览器负责展示与讲解

但现有实现里有三处需要尽快修正，否则“实时展示”和“独立遥测链路”这两个目标都不成立。

## 问题 1：dashboard 默认读取了错误的日志源(已修复)

位置：`dashboard/scripts/run_dashboard.sh:9`

严重级别：P1

当前默认源被改成了 `dashboard/runtime/xv6-console.log`，但这份日志包含的是原始 PTY 控制台输出，不只有 telemetry，还会混入：

- shell 回显
- 用户输入命令
- 普通控制台输出
- prompt 残留

这会直接污染解析结果。当前运行日志里已经能看到典型例子：

- `psSNAP seq=5 ...`
- 下一行变成 telemetry 的续行：`0 ticks=167 ...`

结果是：

- 某些快照会被截断
- 某些样本会被漏掉
- dashboard 展示的数据会静默失真

这不是偶发边角情况，而是你当前主打的“交互 shell 保留可用、后台持续采样”场景下的正常使用路径。

建议：

- 把默认 source 改回 `dashboard/runtime/dashboard-telemetry.log`
- `xv6-console.log` 只保留给调试和原始回放使用
- bridge 默认只消费分离后的 telemetry 文件

修复状态：已修复并完成文档对齐。当前默认链路已经统一为：

- `dashboard/scripts/run_dashboard.sh` 默认读取 `dashboard/runtime/dashboard-telemetry.log`
- `dashboard.bridge.app` 的 `--source` 默认值同步改为 `dashboard/runtime/dashboard-telemetry.log`
- 开发文档已明确 `xv6-console.log` 仅用于调试与原始回放，不再作为 dashboard 默认数据源

## 问题 2：新增的 telemetry side channel 默认根本没有挂上设备(已修复)

位置：`Makefile:241-245`

严重级别：P2

内核现在会在启动时执行：

- `telemetry_console_init()`
- 并探测 `VIRTIO1` / `0x10002000`

但默认 QEMU 启动参数里仍然只有：

- 一个块设备 `virtio-blk-device`

并没有默认附加第二个 virtio telemetry/console 设备。`QEMU_EXTRA_OPTS` 虽然被留了扩展口，但默认值为空，所以在你当前文档化的启动流程里，这条 side channel 实际上不会被启用。

这意味着：

- `telemetrywrite()` 路径默认不可用
- 新加的 `telemetry_console.c` 在默认流程里不会真正工作
- 一旦用户态走 `telemetry_printf`，大概率只会退回 stdout，而不是独立 telemetry 通道

建议：

- 在默认 QEMU 启动参数里显式挂上对应 virtio console / serial 设备
- 或者明确把它定义为“仅在特定 QEMU_EXTRA_OPTS 下启用”的可选能力，并同步修改文档

修复状态：已修复。当前默认启动链路已经补齐并对齐到 side channel 设计：

- `Makefile` 默认 QEMU 参数新增 `virtio-serial-device` 与 `virtconsole`，不再依赖手工补 `QEMU_EXTRA_OPTS`
- `qemu` 和 `qemu-gdb` 目标会自动创建 `dashboard/runtime/`，默认 telemetry 文件为 `dashboard/runtime/dashboard-telemetry.log`
- 默认侧重点仍放在设备挂载本身；现有用户态程序后续可逐步按需切到 `telemetry_printf(...)`

## 问题 3：统一 3 秒刷新不适合任务管理器式实时展示

相关位置：

- `dashboard/js/app.js:1`
- `dashboard/js/app.js:719-730`
- `dashboard/index.html:32`

严重级别：设计问题，建议尽快调整

当前前端把整个 dashboard 绑定在统一的 `REFRESH_MS = 3000` 上，并通过 `setInterval(refreshDashboard, REFRESH_MS)` 定时整页刷新。这种刷新方式适合“讲解型总览面板”，但不适合你现在想做的“任务管理器式实时展示”。

问题非常明确：

- `ls`
- `echo`
- 很多测试前的短命令
- 短生命周期子进程

完全可能在一次 3 秒轮询之间启动并退出。结果就是：

- 进程主表里看不到它们
- 当前运行进程很容易错过
- 用户会误以为系统没反应
- 页面更像慢速状态看板，而不像任务管理器

要注意的是，即使把统一轮询改成 500ms，也仍然不够。因为很多短命命令依然可能在 500ms 内完成。所以“短命令可见性”不能只靠主表轮询保证，而必须靠事件流或短历史缓冲补上。

### 推荐改法：分层刷新

不要让整个 dashboard 用同一个刷新周期。建议拆成三层。

#### 1. 慢速总览层：1 到 3 秒

用于展示趋势型、答辩讲解型信息：

- 当前策略
- ticks
- 总进程数
- RUNNABLE 数
- free pages
- COW / Lazy fault 累计计数
- mmap region 总数

这些字段不要求毫秒级刷新，1 到 3 秒都可以接受。

#### 2. 快速运行态层：200 到 500 ms

用于展示最有“任务管理器感”的内容：

- 进程主表
- 当前 running pid
- state 变化
- queue_level / remaining_time / vruntime
- 当前 CPU 正在跑谁

这部分如果继续保持 3 秒，观感会明显偏慢。

#### 3. 事件层：即时推送或高频差分

这一层专门解决短命令看不见的问题，至少要维护最近事件流，记录：

- fork
- exec
- exit
- scheduler switch
- mmap
- munmap
- lazy fault
- cow fault

这样即使 `ls` 只活了几十毫秒，事件区里仍然能看到它出现过。

### 为什么单靠高频轮询也不够

主表回答的是：

- 现在谁还活着

事件流回答的是：

- 刚刚发生了什么

这两个不能互相替代，必须同时存在。

### 对当前项目更合理的参数建议

方案一，偏答辩展示、实现相对简单：

- summary：1 秒
- process snapshot：500 ms
- event flush：100 到 200 ms
- 图表重绘：500 ms

方案二，偏节制开销：

- summary：2 秒
- process snapshot：1 秒
- event flush：200 ms
- 最近事件保留：10 到 20 秒

### 从 xv6 负载角度的平衡方式

不要直接把所有采样都打到 100ms。更合理的做法是轻重分离：

高频轻量字段：

- seq
- ticks
- running_pid
- runnable_count
- policy
- event counters

低频重量字段：

- 完整进程表
- VMA 列表
- 更详细的内存视图

也就是：

- summary 高频
- full snapshot 低频
- 事件独立推送

## 建议写入设计文档的正式表述

不建议再写：

> 页面每 3 秒刷新一次，实现实时展示。

建议改成：

> 系统采用分层更新机制：总览指标低频刷新，进程状态中频刷新，短时行为通过事件流高频上报，从而兼顾系统开销与短生命周期进程的可观测性。

## 建议的后续修改顺序

1. 先修正默认日志源，确保 bridge 默认消费 `dashboard-telemetry.log`。
2. 再补齐默认 QEMU telemetry 设备挂载，保证 side channel 真正可用。
3. 最后调整前端刷新策略：拆分 summary、process table、event stream 的刷新频率。
4. 如果要进一步增强任务管理器观感，再增加“最近退出进程”或“最近活动事件”区域。

## 总结

模块四目前的问题不是架构方向错了，而是运行链路和刷新策略还停留在“展示型 dashboard”阶段，尚未真正达到“实时任务管理器式可观测性”。

最核心的修正点有三个：

- 默认日志源必须切回干净的 telemetry 文件
- telemetry side channel 必须在默认 QEMU 启动里真实挂载
- 刷新机制必须分层，不能让所有内容统一走 3 秒轮询

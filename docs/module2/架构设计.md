## 1. 模块定位

模块二的核心任务，是把“模块一已经实现若干机制”转化为“这些机制具备可重复验证的正确性证据”。对于本项目而言，测试框架不是附属工具，而是连接内核实现、实验结论和论文论证的质量护栏。

如果没有模块二，模块一的增强只能停留在功能演示层；模块三的实验结果也会失去最基本的正确性前提。因此，模块二在整个系统中的定位，不是额外补几组测试，而是建立一个统一、稳定、可回归的验证平面。

需要说明的是，论文版本的模块二不再把 Buddy 相关测试作为主线叙事的一部分。原因不是仓库中完全没有相关实现，而是模块三没有给出足够能支撑论文结论的效果，因此模块二在论文中聚焦的对象将收敛为：调度、COW、Lazy Allocation、`mmap` 以及这些机制之间的集成路径。

## 2. 为什么要做这个模块

教学操作系统项目最常见的风险，是把“能编译、能启动、能跑几个例子”误认为实现正确。但 xv6 上的调度、懒分配、COW 和 `mmap` 都涉及页表权限、系统调用返回语义、用户态与内核态交互以及时序行为，许多错误并不会立即体现为 panic，而是以更隐蔽的方式污染后续实验和文档。

因此，模块二要解决的并不是“有没有测试”，而是下面几个更本质的问题：

1. 如何把模块一的内核机制转化为可重复执行的断言型验证。
2. 如何避免测试框架自身误解 xv6 语义，从而把正确实现测错。
3. 如何把强断言场景与只能 smoke 的场景分层表述，避免对论文结论过度承诺。
4. 如何在模块三正式实验之前，建立一个足够可信的正确性门禁。

从这个角度看，模块二的价值并不只是“测了多少”，而是“哪些关键结论可以被稳定重复地证明，哪些结论需要诚实保留边界”。

## 3. 模块目标

模块二的设计目标可以概括为以下四点：

1. 为模块一提供统一、稳定的用户态测试入口。
2. 将关键内核机制转化为可重复回归的断言型用例。
3. 区分强断言测试与 smoke 测试，避免过度承诺。
4. 为模块三正式实验提供基本正确性兜底。

## 4. 论文主线中的测试范围

模块二在论文主线中重点覆盖以下几类机制：

1. `mmap/munmap` 的参数、映射与回收行为。
2. COW 的共享、写复制与 `fork+exec` 路径。
3. Lazy Allocation 的申请、访问、缩容与稀疏访问行为。
4. 多机制组合下的集成路径，如 `fork/exec/mmap` 交互。
5. 多种调度策略的切换、顺序、统计与专项 smoke 场景。
6. 资源约束与边界条件下的行为一致性。

仓库中仍存在 Buddy 相关测试，但由于不再服务论文主叙事，后续模块二文档不再把它作为重点覆盖对象展开。

## 5. 测试框架为什么是独立模块，而不是零散测试程序

模块二的核心思想，是把测试从“开发过程中的临时验证脚本”，升级为“系统架构中的正式组成部分”。

如果每个功能只靠单独的用户态程序做一次演示，就会产生三个问题：

1. 测试入口分散，无法形成统一回归集。
2. 测试结果不可枚举、不可批量执行，也不利于定位失败。
3. 模块三实验缺少稳定的前置正确性护栏。

因此，模块二选择单独构建 `tests/` 目录，把测试逻辑与普通用户态程序分离，并通过统一 `test_runner` 管理所有用例。这样做以后，测试不再是附属功能，而是明确进入系统设计的一部分。

## 6. 架构设计：统一测试入口与注册式组织

模块二的统一入口由 [tests/src/test_runner.c](/home/niya/xv6/tests/src/test_runner.c) 提供。`Makefile` 会将该文件与多个测试对象一起链接成统一的 `_test_runner` 程序。

这一架构有两个关键点：

1. 测试源代码与普通用户态程序分离，避免测试代码与实验代码混杂。
2. 所有测试通过统一注册表管理，形成可枚举、可批量运行、可单独定位的回归集。

`test_runner` 支持三种基本调用方式：

1. `test_runner all`
2. `test_runner list`
3. `test_runner <index>`

这意味着模块二既能在日常开发中做局部调试，也能在回归验证中统一执行全部测试。

### 6.1 注册表结构

`test_runner` 通过 `struct test_case tests[]` 统一注册所有用例。该结构定义在 [tests/include/test_common.h](/home/niya/xv6/tests/include/test_common.h)：

```c
struct test_case {
    char *name;
    char *description;
    void (*func)(void);
};
```

这个设计的意义在于：

1. 测试用例可枚举。
2. 总测试数可从注册表自动推导，而不是人工维护。
3. 每条测试都同时具有名称、描述和执行入口，便于输出与定位。

## 7. 测试组织方式：按验证目的分层，而不是只按功能堆叠

从当前实际注册表来看，统一测试框架主要包含以下几类论文主线测试：

| 类别 | 数量 | 关注重点 |
|------|------|----------|
| `mmap` | 7 | 匿名私有映射、参数合法性、映射回收 |
| `COW` | 6 | `fork` 共享、写复制、`fork+exec` 路径 |
| `Lazy` | 10 | 懒分配、访问模式、缩容回收 |
| `Integration` | 10 | `fork/exec/mmap` 等组合路径 |
| `Boundary` | 6 | 资源上限、边界与回收后的再进入 |
| `Scheduler` | 18 | 调度器切换、顺序、统计和专项 smoke |

这种分类不是简单按目录切分，而是按验证目标划分：

1. 功能类测试负责证明单一机制是否工作。
2. Integration 测试负责证明多个机制能否协同。
3. Boundary 测试负责暴露资源约束和边界行为。
4. Scheduler 测试负责覆盖既有强断言场景和无法完全形式化的 smoke 场景。

这种组织方式的价值在于，它直接对应论文里的论证逻辑，而不是把测试写成零散清单。

## 8. 执行流程设计：为什么每条测试都独立 `fork`

测试运行流程如下：

1. `test_runner` 解析参数，决定执行全部测试、列出测试或执行单条测试。
2. 每条测试通过 `fork()` 在子进程中运行。
3. 父进程等待子进程返回，并按 xv6 原生语义解释退出状态。
4. 执行全部测试时，框架汇总通过数和失败数，形成回归摘要。

这种“一条测试一个子进程”的模式有两层意义：

1. 降低不同用例之间的状态污染。
2. 即使单条测试失败，也不会破坏整个测试进程的组织结构。

对操作系统课程项目而言，这种隔离式执行方式尤其重要，因为很多测试本身会改动地址空间、进程树和调度状态。

## 9. xv6 语义适配：为什么测试框架本身也是设计重点

模块二的一个关键经验是，测试框架本身也可能成为误导来源。

例如 [tests/include/test_common.h](/home/niya/xv6/tests/include/test_common.h) 中明确写道：

- `XV6_WAIT_SUCCESS(status)`
- `XV6_WAIT_CODE(status)`

这里的意义在于，xv6 的 `wait()` 返回语义与常见类 Unix 用户态库中对状态字的处理习惯并不完全相同。如果直接照搬 POSIX 宏，测试就可能把正确行为判错，或者把错误行为误判为成功。

因此，模块二并不只是“测内核”，它也在不断修正“测试对 xv6 的理解方式”，保证验证逻辑本身是可信的。

## 10. 补充测试入口：为什么还保留专项程序

除了统一 `_test_runner` 外，仓库中还保留了若干专项程序，例如：

- `scheduler_test`
- `mmap_test`
- `test_sjf`
- `test_srtf`
- `test_mlfq`
- `test_cfs`
- `test_stats`
- `test_snapshot`

这些程序承担的是“聚焦单一主题、便于动态观察”的角色。它们不替代统一测试框架，而是与 `_test_runner` 一起构成模块一修复后的最小回归集。

换句话说，模块二不是只追求一个统一入口，而是在统一入口之外，保留少量便于调试和展示的专项入口，从而兼顾回归与观察两种需求。

## 11. 关键设计取舍

### 11.1 为什么强调“强断言”和 “smoke” 的分层

调度算法尤其容易出现一个误区：只要进程最终跑完，就被当作算法正确。事实上，LOTTERY、SJF、SRTF、MLFQ 和 CFS 的许多语义需要更丰富的用户态观测点才能彻底证明。在当前 xv6 环境下，并非所有高级调度场景都能通过用户态完全形式化验证。

因此，模块二明确区分两类测试：

1. 强断言测试：例如 FCFS 顺序、PRIORITY 顺序、部分边界行为和关键内存语义。
2. smoke 测试：例如策略可切换、能运行、统计值与元数据保持合理。

这种分层不是弱化测试，而是为了保证论文表述诚实，不把尚无充分观测支持的行为写成“已完全证明”。

### 11.2 为什么强调 xv6 原生语义，而不是照搬 POSIX 习惯

`wait()` 返回值解释错误就是典型例子。如果测试框架对 xv6 的系统调用语义理解本身就不准确，那么测试越多，误导也越严重。

因此，模块二的原则是：测试必须服从被测系统的真实语义，而不是开发者的先验习惯。

### 11.3 为什么模块二不追求覆盖率数字，而强调关键路径

对于教学操作系统项目而言，更重要的是覆盖那些最容易让论文结论失真的路径，而不是机械追求一个覆盖率数字。

模块二真正重视的是：

1. COW 是否同时覆盖共享、写复制和 `fork+exec`。
2. Lazy Allocation 是否覆盖访问模式变化与缩容行为。
3. `mmap/munmap` 是否覆盖参数校验和回收边界。
4. 调度器切换后系统是否还能稳定运行并维持关键元数据。
5. 多机制组合时，局部修复是否破坏整体行为。

## 12. Bug 如何推动方案收敛

模块二的最终形态，同样是通过若干关键问题不断收敛出来的。

### 12.1 测试数量硬编码问题

当测试总数依赖人工维护时，一旦新增或删除测试，用例统计就会失真。这说明测试框架自身也可能引入维护风险。

修复后，测试总数统一由注册表推导，从而减少了框架层面的伪一致性问题。

### 12.2 `exit_status` 处理错误

该问题说明，把外部系统习惯直接套到 xv6 上，会导致误测。修复后，`test_runner` 明确按 xv6 原生 `wait()` 语义解释结果，从而避免把框架错误误当成功能错误。

### 12.3 调度器名称与观测层不同步

该问题说明，观测层与实现层必须同步更新，否则测试输出与系统真实状态会脱节。这类问题与模块四中的监控同步问题本质相同，都是“观测模型落后于内核实现”。

### 12.4 Lazy 测试路径不真实

当 Lazy 测试没有真正走到 `sbrklazy()` 路径时，测试名义上在测 Lazy，实际上却没有验证目标机制。整改后，测试才真正对准被测对象。

## 13. 审查与整改后形成的最终口径

结合当前实现与整改结果，模块二可以给出如下更准确的判断：

1. 已建立统一入口和分类清晰的回归集。
2. 关键误测点已经修正。
3. 多个核心内存机制具备较强断言支撑。
4. 高级调度器中的部分复杂语义仍需结合专项测试和实验观察综合判断。
5. Buddy 相关测试不再作为论文结论的重要支撑点。

### 13.1 已确认完成的整改

1. `test_runner` 已按 xv6 原生退出码语义判断测试结果。
2. Lazy 测试已改为真实调用 `sbrklazy()`。
3. FCFS、PRIORITY、PRIORITY-FCFS 已加入同步起跑与顺序断言。
4. LOTTERY 已明确收敛为 smoke，不再假装验证权重分布。
5. COW 的读共享、写复制和 `fork+exec` 路径断言已收紧。

### 13.2 仍需诚实保留的限制

1. `SJF/SRTF/MLFQ/CFS` 的深层调度语义仍缺少更强观测点。
2. 模块二的定位是回归保护层，不是替代模块三实验方法学。
3. 模块二证明的是“工程上足够可信的行为正确性”，不是形式化验证。

## 14. 验证闭环：模块二如何服务整个系统

模块二在整个项目中的闭环关系可以概括为：

模块一实现机制 -> 模块二把机制转化为统一回归测试 -> 模块二先排除明显功能错误和误测风险 -> 模块三再在正确性前提上做正式实验 -> 模块四展示和监控系统状态。

这意味着模块二既不是模块一的附录，也不是模块三的替代品，而是两者之间的质量门禁。

更具体地说，它承担了三层作用：

1. 对模块一做功能正确性兜底。
2. 为模块三提供实验前的可信前提。
3. 为论文答辩提供“结论不是拍脑袋得出”的工程证据链。

## 15. 最终结论

模块二已经不再是“堆很多测试用例”的辅助部分，而是形成了一个统一、稳定、分层清晰的回归验证模块。

从架构角度看，它完成了三件关键事情：

1. 建立了统一测试入口和注册式组织方式。
2. 建立了从强断言到 smoke 的分层验证策略。
3. 把模块一的实现结果转化为模块三实验可以依赖的正确性前提。

因此，更准确的论文表述应当是：模块二通过统一测试框架和分层验证策略，为模块一提供工程上足够可信的正确性证据，并为模块三实验建立质量门禁，而不是简单以“测试数量多”来证明系统实现充分。

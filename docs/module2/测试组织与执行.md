# 测试组织与执行

模块二这部分只看“测试是怎么被组织起来的、怎么被执行起来的”。

它覆盖三个最关键的点：

1. `test_runner` 为什么是统一入口。
2. `struct test_case` 为什么要注册式组织。
3. 为什么每条测试都独立 `fork()`。

更完整的背景和边界仍然保留在：

- [模块二架构设计](/home/niya/xv6/docs/module2/架构设计.md)

## 统一入口

`test_runner` 把测试源和普通用户态程序分开，避免测试代码和实验代码混在一起。

它支持三种常见模式：

- `test_runner all`
- `test_runner list`
- `test_runner <index>`

## 注册式组织

`struct test_case` 把每个用例的名称、描述和函数入口都注册起来。这样做的好处是：

- 测试数量可以自动推导。
- 用例可以单独定位。
- 批量执行和局部调试都方便。

## 独立 fork

每个测试单独在子进程跑，是为了：

- 降低状态污染。
- 让失败用例不拖垮整个回归集。
- 适配 xv6 里地址空间、进程树和调度状态很容易互相影响的现实。

- 调度模块形成了完整的 `sched_ops + 生命周期钩子` 架构闭环。
- 内存模块形成了完整的 `Buddy + COW + Lazy + mmap + 统一 vmfault` 架构闭环。
- 原先最影响论文可信度的几处核心断点，已经在当前版本中修复并接线完成。

因此，和旧版审查相比，当前结论不再是“有 6 个关键问题尚未闭环”，而是：

- 主架构已经成立，代码设计基本正确。
- 仍有若干“应当诚实表述为简化实现”的边界与剩余风险。
- 对于本科毕设，这个完成度已经足够支撑架构说明、实验设计和答辩展示。

## 二、相对旧版审查，已经确认修复的关键问题

### 2.1 调度器生命周期钩子已经真正接入内核路径

当前版本中：

- `allocproc()` 会调用 `sched_proc_init_hook(p)`。
- `kexit()` 会调用 `sched_proc_exit_hook(p)`。
- `sched_update_stats()` 会在运行进程上调用 `sched_proc_tick_hook(current)`。
- `wakeup()` 会调用 `sched_proc_wakeup_hook(p)`。
- `sched_set_policy()` 在切换策略前会对整套策略状态和当前所有活动进程做一次 `sched_init_policy_state()`。

这意味着 SJF/SRTF/CFS 这类依赖私有状态数组的调度器，已经不再处于“定义了接口但没有接线”的状态。

### 2.2 SRTF 已经具备剩余时间递减逻辑

当前 `SRTF` 已经做到：

- 为每个进程维护 `remaining_time[]`。
- 在 `proc_tick` 钩子里每个 tick 递减一次。
- 在选择进程时优先选择剩余时间更短者。
- 当剩余时间相同，使用 `ctime` 进行稳定 tie-break。

因此，旧版“remaining time 根本不变”的问题已被修复。

### 2.3 MLFQ 已经接上反馈机制的两个核心钩子

当前 `MLFQ` 已经具备：

- 时间片耗尽后降级。
- 睡眠后被唤醒时提升一级。
- 队列内 RR 轮转。

所以它已经不是“纯扫描顺序上的 RR 伪装版 MLFQ”。

需要保留的说明是：当前还没有周期性全局 boost，因此它应表述为“简化版 MLFQ”。

### 2.4 `mmap/munmap` 已有最小可用的 VMA 记录机制

当前版本中：

- `struct proc` 增加了 `heap_end` 和 `vmas[NVMA]`。
- `sys_mmap()` 会登记 VMA，而不是只增加 `p->sz`。
- `sys_munmap()` 支持完整释放、头部释放、尾部释放和中间拆分。
- `proc_vm_top()` 会重新计算新的地址空间上界。
- `vmfault()` 会先判断地址是否落在合法 VMA 中。

因此，旧版“`munmap` 后空洞会重新被懒分配补出来”的主问题已被修复。

### 2.5 `mmap` 权限已经落到真实页表权限上

当前 `vmfault()` 通过：

- `vmfault_region()` 判定地址类型和访问合法性。
- `vma_prot_to_perm()` 把 `PROT_READ / PROT_WRITE` 转成对应的 PTE 权限。

这意味着：

- `PROT_READ` 映射不再默认变成可写页。
- 写只读匿名映射会在 fault 时被拒绝。

因此，旧版“`prot` 只是摆设”的问题已被修复。

### 2.6 CFS 的 `min_vruntime` 初始化和继承逻辑已修正

当前 `CFS` 会：

- 在新进程初始化时扫描现有活动进程，找到当前系统的最小 `vruntime`。
- 让新进程继承这个最小值，而不是永远从 0 起跑。
- 在进程退出时重新计算系统的 `min_vruntime`。

因此，旧版“新任务永远凭 0 值插队”的问题已被修复。

## 三、当前架构是否正确

### 3.1 调度架构评价

当前调度模块的架构是合理的，核心优点有三点：

1. **策略解耦清晰**：`pick_next()` 和私有状态更新被拆开，不同算法可以只关心自己的选择逻辑和状态数组。
2. **接线集中可审计**：所有关键生命周期入口都能在 `proc.c / trap.c / sched_stats.c` 中找到统一接入点。
3. **可扩展性较好**：新增算法主要在 `kernel/sched/` 内部完成，不需要到处改主内核路径。

对于 xv6 这种小型教学内核来说，这套设计已经比“在 `scheduler()` 里直接写一大坨 `if(policy == ...)`”高一个层次，足够支撑毕设中的架构设计部分。

### 3.2 内存架构评价

当前内存模块的架构同样是成立的，关键在于它不再是四个孤立 feature，而是统一到了以下主线上：

- `kalloc.c` 负责物理页与统计。
- `proc.{h,c}` 负责进程地址空间元数据。
- `vm.c` 负责真正的页表复制、缺页处理和权限建立。
- `sysproc.c` 负责把用户态接口翻译为内核地址空间操作。
- `trap.c` 负责把 fault 路径导向 COW 或统一的 `vmfault()`。

特别是 `heap_end + vmas[] + sz` 这个地址空间模型，是当前版本中最值得在论文里强调的架构点。

## 四、当前仍需诚实说明的限制

这部分不是“bug 清单”，而是“论文和答辩时不能说过头的地方”。

### 4.1 SJF/SRTF 仍属于教学化估计模型

当前 SJF/SRTF 的估计逻辑基于历史运行时间的简化 EWMA，而不是严格意义上的 CPU burst 观测模型。因此更适合表述为：

- “基于历史运行时间估计的 SJF/SRTF 原型实现”

而不建议直接写成：

- “完整复刻经典教科书级 SJF/SRTF”

### 4.2 MLFQ 仍缺少周期性全局提升

当前 MLFQ 已有“时间片降级 + 唤醒提升”，但没有周期性全局 boost。答辩时若老师问到“如何避免底层队列长期饥饿”，应直接说明当前是简化版实现。

### 4.3 `mmap` 是匿名私有懒映射原型，不是完整 POSIX `mmap`

当前支持：

- `MAP_PRIVATE`
- `MAP_ANONYMOUS`
- `PROT_READ/PROT_WRITE`
- `munmap` 子区间拆分

当前不支持：

- 文件映射
- `MAP_SHARED`
- 固定地址映射
- 脏页回写

因此论文中最稳妥的说法是：

- “实现了匿名私有懒映射的教学化原型”

### 4.4 Buddy 的 buddy 查找仍是链表扫描

这会影响理论上的最优复杂度表达，但不影响当前功能正确性和实验可用性。可作为后续优化点，而不必再当作主缺陷。

### 4.5 动态验证已经通过，但断言强度仍以教学测试为主

当前通过的测试足以证明：

- 代码能编译、能启动、能切换调度器、能跑基本内存路径。
- 关键 feature 已经接线。

但这并不等同于“每个高级算法都已经被数学上严格验证”。论文中建议如实写成“基于功能测试与实验验证”。

## 五、建议在论文中强调的架构亮点

如果要突出模块一的工程设计价值，我建议重点写下面 4 个点：

1. **统一调度框架**：通过 `sched_ops` 与生命周期钩子，把 9 种调度器纳入同一可切换框架。
2. **统一缺页路径**：通过 `vmfault()` 把 Lazy Allocation、`mmap` 权限建立和部分系统调用路径统一起来。
3. **地址空间元数据升级**：用 `heap_end + vmas[] + sz` 替代原始 xv6 单一 `sz` 语义，使 `munmap` 与懒分配能够共存。
4. **统计数据出口统一**：调度统计与内存统计都提供了用户态可读取接口，为模块二测试、模块三实验和模块四监控复用。

## 七、最终建议

当前模块一已经不再适合沿用旧版“问题清单式否定结论”。更合适的结论是：

- **设计上正确**：核心架构已经闭环。
- **工程上可用**：代码可以编译、启动、运行、测试。
- **论文上可写**：可以作为正文和答辩依据，但应明确写成“教学化、简化实现”。
- **后续仍可提升**：重点优化方向是 MLFQ 全局 boost、SJF/SRTF 更细粒度 burst 估计，以及 Buddy 的更优空闲块索引结构。

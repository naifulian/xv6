# 模块四审查记录

审查时间：2026-03-26

## 审查结论

模块四目前已经不是“全是下拉菜单”的早期状态，但它离你真正要的“xv6 教学内核任务管理器”还有明显距离。问题不只是页面不好看，而是产品形态、数据协议、启动链和运行链路都还停留在“工程样机”阶段：

1. 前端还是单页拼盘，不是“主界面 -> 子界面 -> 返回主界面”的任务管理器式结构。
2. 启动链仍然要求用户手工开两个终端协作，不符合“一条命令拉起监控器”的目标。
3. 仓库里已经有独立 telemetry side channel，但 `dashboardd` 仍然走 `stdout`，默认链路还依赖 `split_console.py` 做 PTY 文本分流。
4. 当前真正有论文价值的页面主线应收敛到进程、调度、内存与事件，不应再额外扩展无关页面。

因此，模块四接下来的正确方向不是继续堆单页卡片，而是先把目标架构定准，再按架构收口代码。

## 先定目标架构

### 1. 产品定位

模块四应该被定义为：

> 一个面向 xv6 教学场景的 Web 任务管理器，而不是实验结果网页、也不是单页调试面板。

它的核心职责应当是把 xv6 的运行态拆成几个可以独立理解的教学视图，而不是把所有内容塞进一个大页面里让用户自己找。

### 2. 前端信息架构

建议采用“主界面 + 子界面”的结构，而不是继续维护一个超大的 `index.html`。

推荐页面组织如下：

1. `主页 / Home`
   - 展示模块入口卡片：进程管理、调度观察、内存管理、事件时间线。
   - 每张卡片只放 2 到 4 个高价值实时摘要，例如 `RUNNABLE`、`free pages`、当前策略、最近事件数。
   - 点击卡片进入子界面。

2. `进程管理页 / Processes`
   - 左侧主表，右侧选中进程详情。
   - 保留 PID、state、priority、tickets、rutime、retime、stime、sz。
   - 详情区展示时间分解、调度解释、地址空间摘要、VMA 列表。
   - 顶部必须有“返回主页”。

3. `调度观察页 / Scheduler`
   - 不和进程主表混在同一屏里。
   - 每种策略展示自己的教学重点：
     - FCFS 看长作业占用。
     - PRIORITY/SML 看 priority 分布。
     - LOTTERY 看 tickets 集中度。
     - SRTF 看 remaining time。
     - MLFQ 看 queue level。
     - CFS 看 vruntime。

4. `内存管理页 / Memory`
   - 系统级视图：`total/free/used pages`、`cow faults`、`lazy faults`、`mmap regions`。
   - 进程级视图：`sz`、`heap_end`、VMA 列表。
   - 必须明确口径：`sz` 只是逻辑地址空间规模，不是假装 RSS。

5. `事件时间线页 / Events`
   - 专门看 `fork/exec/exit/scheduler switch/mmap/munmap/cow/lazy fault`。
   - 与进程主表分离，解决短命命令“在主表看不见”的问题。

### 2.5 答辩展示性与教学展示性的页面原则

模块四不应该只有一个“监控模式”，而应该同时服务两个场景：

1. `答辩导览模式 / Defense Mode`
   - 首页按四个模块组织内容，而不是一上来就铺满监控卡片。
   - 页面叙事顺序应当是：研究问题 -> 模块一机制设计 -> 模块二验证证据 -> 模块三实验结论 -> 模块四现场演示。
   - 每张模块卡都只展示最能代表论文贡献的内容：
     - 模块一：9 种调度策略、统一调度框架、统一地址空间语义。
     - 模块二：测试分类、关键回归通过、实现边界已经被验证。
     - 模块三：正式实验结论、代表性图表、哪些结论成立、哪些结论需要保留边界。
     - 模块四：实时运行态入口，负责把前面三个模块“讲活”。
   - 这个模式下的页面目标不是“信息最全”，而是“让评委顺着你的论文主线走”。

2. `教学观察模式 / Teaching Mode`
   - 面向课堂、演示和日常观察，直接进入进程、调度、内存、事件几个子页面。
   - 每个子页面都要稳定回答三个问题：
     - 现在看到的是什么状态。
     - 它对应模块一里的哪个机制。
     - 为什么这个指标值得学生或评委关注。
   - 因此每个子页面建议固定保留三类辅助区域：
     - `机制说明`：解释当前页在讲什么系统机制。
     - `观测口径`：说明字段边界，避免把 `sz` 之类的教学口径说成 Linux 指标。
     - `建议演示命令`：告诉答辩时该在 xv6 里敲什么命令来触发这个页面的变化。

3. `答辩与教学的共同要求`
   - 首页不应只是“Dashboard 首页”，而应是一张“论文闭环图 + 运行态入口图”。
   - 页面必须把四个模块的关系讲清楚：模块一给机制，模块二给可信性，模块三给正式证据，模块四给展示平面。
   - 页面必须诚实，不伪造 Linux 级指标，不把教学 OS 没有的数据包装成通用系统监控指标。
   - 页面必须支持从论文导览跳到运行态页面，再从运行态页面返回论文导览，否则它只适合演示，不适合答辩。

从毕设主题出发，模块四最有价值的首页其实不是“当前 CPU/内存有多少”，而是“这套系统为什么值得展示、这四个模块如何形成闭环、当前 live monitor 能帮你现场证明什么”。

实现方式建议：

- 页面层采用 `shell + router + view modules`，而不是继续把全部 UI 逻辑塞进一个 `app.js`。
- 路由可以先用轻量方案，例如 `/#/home`、`/#/processes`、`/#/scheduler`、`/#/memory`、`/#/events`。
- 每个子页面只订阅自己需要的数据，不要所有页面都强绑定在一个全量 payload 上。
- 启动说明、调试命令、日志路径不应占据主界面主视觉区，它们应移到帮助弹层或开发文档。

### 3. 数据架构

前端想拆成多页面，前提是数据面也跟着拆。建议把模块四的数据层正式分成五类：

1. `summary`
   - 总览卡片和主页用。

2. `process snapshot`
   - 进程主表与选中进程详情用。

3. `scheduler view`
   - 按策略解释字段，而不是只复用进程表。

4. `memory view`
   - 系统内存状态和选中进程地址空间用。

5. `event view`
   - 最近事件、短命命令可见性和状态变化解释用。

建议的数据协议演进：

1. 保留现有 `SNAP / PROC / VMA`。
2. 新增 `EVENT` 帧，避免事件完全依赖快照差分。
3. 在 `pstat` 或专用 syscall 中补齐真正决定页面质量的字段：`running_pid`、`queue_level`、`remaining_time`、`vruntime`。
4. 新增 `EVENT` 帧之外的聚合摘要字段，减少前端对全量快照差分的依赖。

如果不想一次性加太多 syscall，建议新增一个聚合接口，例如：

- `getdashframe()`：返回主页/总览/调度/内存需要的聚合摘要。
- `getprocvmas(pid, ...)`：保留按需详情。

### 4. 启动架构

正确的启动体验应该是：

> 一条命令启动模块四，然后当前终端直接进入 xv6 交互；Web bridge 在后台自动起来；用户只需要打开浏览器。

推荐目标流程：

```text
dashboard/scripts/start_monitor.sh
-> 创建 runtime 目录与日志文件
-> 后台启动 bridge
-> 当前终端前台启动 xv6/QEMU
-> bridge 自动消费 telemetry 文件
-> 浏览器访问 dashboard
-> 退出 xv6 时自动清理 bridge
```

启动脚本应满足这些约束：

1. 当前终端保留给 xv6 交互，不再要求用户额外手工开第二个终端。
2. bridge 由脚本后台拉起，并通过 `trap` 自动清理。
3. 如果存在图形桌面或 `tmux`，可以再提供“新终端/新 pane 打开日志”的增强模式，但这只能是增强项，不应成为默认依赖。
4. 默认链路优先使用独立 telemetry side channel；`split_console.py` 只作为 fallback/debug 工具。

建议分两步落地：

1. 先做“统一启动脚本”，哪怕底层仍暂时复用 `split_console.py`。
2. 再把 `dashboardd` 从 `printf(...)` 切换到 `telemetry_printf(...)`，让独立 virtconsole 真的成为默认路径。

一旦第二步完成，默认启动逻辑可以简化成：

```text
start_monitor.sh
-> 后台启动 bridge
-> 前台执行 make qemu
-> dashboardd 直接把遥测写入 telemetry virtconsole
-> bridge 直接读取 dashboard-telemetry.log
```

这样就不再需要对 PTY 控制台输出做启发式拆分，交互体验和稳定性都会明显好很多。

## 当前代码审查结果

下面的结论基于当前仓库代码，而不是基于旧文档假设。

### P1：前端仍然是单页拼盘，和“主界面 -> 子界面”的目标不一致

位置：`dashboard/index.html:12-212`，`dashboard/js/app.js:1-990`

当前页面把主页、进程、调度、内存、趋势、事件、启动说明全部堆在一个页面里。代码里也没有真正的路由、页面状态机、返回主页按钮或页面级渲染边界，`app.js` 是一个接近千行的全局脚本，负责所有视图的刷新与 DOM 拼装。

这会带来三个直接后果：

1. 你想要的“主页点击进入子页面，再返回主页”根本无处安放。
2. 页面职责继续增加时，`index.html + app.js` 会越来越难维护。
3. 进程、内存、事件、启动说明被混在同一个信息平面里，产品形态更像 demo 大屏，不像教学任务管理器。

建议：优先做页面级拆分，而不是继续在当前单页里补卡片和区块。

### P1：启动链仍然要求两个终端，违背“一条命令拉起监控器”的目标

位置：`dashboard/scripts/run_dashboard.sh:21-34`，`dashboard/scripts/capture_xv6_console.sh:14-30`

`run_dashboard.sh` 现在只负责起 bridge，并明确提示用户再去另一个终端运行 `capture_xv6_console.sh`。这说明当前启动模型本质上还是“桥接服务脚本”和“xv6 交互脚本”两套手工拼接流程，而不是一个完整的监控器入口。

这不是体验小问题，而是架构没有收口：

1. 用户必须记住两条命令。
2. bridge 生命周期和 xv6 生命周期分离，没有统一托管与清理。
3. 页面主界面还把“Terminal A / Terminal B”写进了产品 UI，本身也说明当前实现仍处于开发态。

建议：新增统一入口脚本，例如 `dashboard/scripts/start_monitor.sh`，由它负责后台启动 bridge、前台进入 xv6、退出时自动清理。

### P1：独立 telemetry 通道已经具备，但默认实现仍然走 `stdout`，导致默认链路继续依赖 `split_console.py`

位置：`Makefile:245-248`，`kernel/sysproc.c:47-71`，`user/telemetry_printf.c:14-216`，`user/dashboardd.c:59-110`，`user/sysmon.c:54-93`

当前仓库里其实已经有完整的独立 telemetry 基础设施：

1. `Makefile` 默认挂载了 `virtio-serial-device` 与 `virtconsole`。
2. 内核里有 `telemetrywrite()` syscall。
3. 用户态里有 `telemetry_printf()` 封装。

但真正负责发遥测的 `dashboardd` 和 `sysmon` 仍然全部使用 `printf(...)` 输出到标准控制台。结果是：

1. 默认链路仍然污染交互终端。
2. 默认运行仍然要靠 `split_console.py` 从 PTY 文本里猜哪些行属于 telemetry。
3. “独立 side channel” 在架构上存在，在默认产品路径里却没有被真正用起来。

这就是当前启动体验迟迟收不拢的根因之一。只要发射端不切到 `telemetry_printf(...)`，双终端/文本分流方案就始终是默认路径，而不是临时兼容层。

建议：把 `dashboardd` 默认输出改到 `telemetry_printf(...)`，把 `split_console.py` 降级为调试备用方案。

### P1：模块四页面范围应主动收敛，不要偏离论文主线

位置：`docs/开发进度.md:8-16`，`docs/module1/架构设计.md:1-37`，`docs/module3/架构设计.md:1-54`，`dashboard/index.html:12-212`

从当前项目口径看，模块四的主要使命是把模块一的运行机制、模块二的验证可信性和模块三的正式结论组织成一个可答辩、可教学的 live monitor。这个主线已经足够重了，因此页面范围应该主动收敛到：

1. 主页 / 答辩导览
2. 进程页
3. 调度页
4. 内存页
5. 事件页

继续扩展“磁盘页面”这种不在当前论文主线中的内容，只会带来三个问题：

1. 稀释答辩时的叙事重点。
2. 增加新的数据协议和实现成本。
3. 把模块四从“为毕设主题服务”拉回“做一个什么都想展示的 dashboard”。

建议：文档、页面导航和后续开发计划都统一去掉磁盘页面，把资源集中在进程、调度、内存和事件四条主线。

### P2：bridge 在日志变化时全量重读并重算，前端又做多路轮询，页面继续扩展后会越来越重

位置：`dashboard/bridge/app.py:39-46`，`dashboard/bridge/state.py:19-33`，`dashboard/bridge/parser.py:108-142`，`dashboard/bridge/parser.py:505-584`，`dashboard/js/app.js:961-980`

当前 bridge 的工作方式是：只要 telemetry 文件的 `mtime` 变了，就重新 `read_text()` 整个文件，再重新解析所有 `SNAP/PROC/VMA`，再重新做一次事件差分。与此同时，前端还在用三路轮询：

- overview：1s
- process：0.5s
- events：0.2s

这意味着只要日志持续增长，bridge 就会被反复触发做“全量重算”。在当前数据量小的时候问题不明显，但一旦你把页面拆成更多子页面、把采样时长拉长、或把事件做得更丰富，这个设计会先把 bridge 自己拖慢。

建议：

1. bridge 改成增量 tail/增量解析，而不是每次全文件重读。
2. 前端改成页面级订阅，只拉当前页面真正需要的数据。
3. 条件允许时改成 WebSocket 或 SSE，降低重复 REST 拉取。

### P2：当前“事件流”只是快照差分，不能保证短命命令可见

位置：`user/dashboardd.c:44-46`，`user/dashboardd.c:62-107`，`dashboard/bridge/parser.py:505-554`

当前事件流不是 guest 侧主动上报的真实事件，而是 bridge 通过相邻快照做差得到的“派生事件”。这会有一个天然上限：

如果某个短命进程在两次采样之间创建并退出，那么它既可能不出现在主表，也可能不出现在事件流里。

这和“任务管理器式可观测性”是有差距的，尤其是你后面想演示 `ls`、`echo`、测试程序这类很短的命令时，页面会显得迟钝甚至像“没反应”。

建议：

1. 保留快照差分事件作为兜底。
2. 进一步增加独立 `EVENT` 帧，至少覆盖 `fork/exec/exit/scheduler switch/mmap/munmap`。

### P3：启动提示与实际默认参数已经出现文案漂移

位置：`dashboard/scripts/run_dashboard.sh:29`，`user/init.c:13`

`run_dashboard.sh` 里写的是 `dashboardd 30 0`，但 `init.c` 当前实际自动拉起的是 `dashboardd 10 0`。这不是大问题，但它说明启动说明、页面文案和真实代码已经开始漂移。模块四后续如果继续迭代 UI 和启动链，这类不一致会越来越多。

建议：把启动说明从页面主内容里拿掉，并让脚本直接打印真实参数，而不是手写常量文案。

## 建议整改顺序

建议按下面顺序收口，而不是同时乱改：

1. 先做统一启动脚本，解决“双终端手工协作”的体验问题。
2. 再把 `dashboardd` 默认输出切到 `telemetry_printf(...)`，让独立 telemetry side channel 真正成为默认链路。
3. 然后拆前端为“主页 + 子页面”，把当前单页拼盘改成真正的任务管理器式路由结构。
4. 接着补 `EVENT` 帧和页面级订阅，解决短命命令和高频交互的可观测性问题。
5. 最后把首页导览、进程、调度、内存、事件五个页面做成稳定可答辩版本。

## 最终判断

模块四现在最需要的不是继续微调当前页面，而是完成一次真正的“产品化收口”：

- 从单页调试面板，收口成主界面 + 子界面的教学任务管理器。
- 从双终端手工启动，收口成一条命令自动拉起。
- 从 `stdout + split_console.py` 过渡方案，收口成默认 side channel。
- 从“单页堆叠”，收口成可支撑主页、进程、调度、内存、事件几个子页面的完整协议。

在这个目标架构下，当前代码最值得优先改的地方不是颜色和排版，而是页面结构、启动结构和遥测结构。

### Findings

#### P0：telemetry virtconsole 写路径仍会重帧/丢帧，当前“重复进程列表”的根因在 guest -> host 数据链，而不在前端表格

位置：`kernel/telemetry_console.c:109-136`，`dashboard/bridge/parser.py:163-167`，`dashboard/bridge/parser.py:562-569`

这一轮最关键的发现，是“进程列表偶尔出现两个同样的进程”并不是纯前端问题，而是底层 telemetry 已经被写坏了。

当前 `telemetry_console_write()` 每次发送都复用同一个 `txbuf` 和同一个描述符 `desc[0]`，但并没有在下一次写入前确认这一包已经被设备真正消费完成。这样在 `dashboardd` 一次采样里连续输出 `SNAP`、多个 `PROC`、多个 `VMA` 时，就很容易出现：

1. 前一帧还没被 host 取走，缓冲区就被后一帧覆盖。
2. host 侧日志出现同一 `(seq, pid)` 的重复 `PROC` 记录。
3. 某些 `SNAP` 帧直接缺失，导致同一批 `PROC` 没有对应系统快照。

这不是推测。对当前 `dashboard/runtime/dashboard-telemetry.log` 做现场统计时，已经能看到：

1. 存在重复 4 次的 `(seq, pid)` 组合，例如 `163:3`、`158:3`。
2. 统计到 `90` 个只看到了 `PROC`、没有看到对应 `SNAP` 的 `seq`。

而 host parser 现在又会把同一 `seq` 下的 `PROC` 直接 append 到列表里，不做 `(seq, pid)` 去重；同时 `latest_seq` 只取最后一个 `SNAP` 的 `seq`。于是结果就是：

1. 页面会直接展示重复进程。
2. 即使最新日志里已经有新的 `PROC`，页面也可能因为 `SNAP` 丢了而停在更早的那一批状态。

这条问题如果不先解决，模块四前端再怎么调布局，展示结果都不可信。

建议修复顺序：

1. 根修：把 `telemetry_console_write()` 改成真正可靠的 virtqueue 发送模型，至少保证“缓冲区所有权交还之前不复用同一包数据”。
2. 兜底：host parser 对同一 `(seq, pid)` 做 last-write-wins 去重，别让一条坏 telemetry 直接污染整个进程表。
3. 健壮性：当 `PROC` 和 `SNAP` 不配对时，要么丢弃这批不完整样本，要么把它们标记为 degraded，而不是默认当成正常实时数据。

这也是“如何解决页面里两个同样进程”的最直接答案：**先修 guest 侧写链路，再在 host 侧做去重兜底。**

#### P1：`ps` 仍把 64 项 `pstat` 表放在用户栈上，当前镜像里可以直接复现 `vmfault`

位置：`user/ps.c:17-34`

这条问题已经做了现场验证，而不是只停留在代码阅读。

我在当前仓库直接执行：

```text
make CPUS=1 qemu
```

进入 shell 后，执行：

```text
ps
```

现场结果是：

```text
usertrap(): vmfault failed pid=4 va=0x2e08
```

根因也很直接：`user/ps.c` 仍然在用户栈上放了 `struct pstat ps[64]`。当前 `struct pstat` 已经不是很小的结构体，`64` 项表本身就会逼近甚至超过 xv6 一页用户栈的可承受范围。模块四这轮已经在 `dashboardd/sysmon` 里把大缓冲区改成静态存储了，但 `ps` 还保留着旧写法，所以它正好成了“启动后部分命令仍然报错”的现场复现点。

这条问题和模块四虽然不是同一个文件夹里的代码，但它直接影响模块四演示可信度：答辩时如果你在 xv6 里敲一个最基础的 `ps` 都炸掉，评委不会先去分辨“这是用户工具 bug，不是内核 bug”。

建议：

1. 把 `ps` 的 `struct pstat ps[64]` 改成静态存储，或者分批拉取。
2. 顺手审一遍其它用户工具里类似的大数组，至少把 `perftest.c:529` 这类同型风险一起清掉。
3. 把 `ps`、`memstat`、`ls` 纳入模块四启动后的最小 smoke test。

#### P1：进程页、调度页和 `ps` 工具把 `sched_class` 当成“当前调度策略”展示，参数口径已经偏离模块一真实语义

位置：`kernel/proc.c:180-189`，`user/ps.c:44-59`，`user/dashboardd.c:86-92`，`user/sysmon.c:80-86`，`kernel/sysinfo.c:65-76`

这一条会直接影响你问的“Web 界面展示的参数是否合理”。

当前代码里：

1. `sys_snapshot` 的 `sched_name` 来自 `current_policy`，这表示系统当前启用的全局调度策略。
2. 但 `PROC` 和 `ps` 又把 `p->sched_class` 映射成 `DEFAULT/FCFS/PRIORITY/...` 这些策略名。
3. 同时 `allocproc()` 里把 `p->sched_class` 默认初始化成 `DEFAULT_SML_CLASS`，其值是 `2`。

于是，当前系统很容易出现这种自相矛盾的展示：

1. `SNAP sched=DEFAULT`
2. `PROC sched=PRIORITY`

这不是“页面显示风格不同”，而是字段语义错了。`sched_class` 在当前实现里更像是 SML 相关的进程分类字段，不等价于系统正在运行的全局策略；把它直接当成“当前 scheduler 名称”展示，会让前端、`ps`、日志三边都出现误导。

如果模块四要服务模块一答辩，这条问题必须修。因为模块一强调的是“统一调度框架 + 9 种策略切换”，而不是“每个进程都各自运行在不同的调度器上”。

建议：

1. 进程级字段和系统级字段彻底分开。
2. `policy` 只表示全局当前策略，来自 `current_policy`。
3. `sched_class` 如果保留，就改名成更不易误解的字段，例如 `sml_class` / `queue_class`，并只在对应策略下解释。
4. `ps` 的 `-s` 过滤也要同步改口径，否则它现在过滤的不是“scheduler”，而是某个进程级类别值。

#### P1：`exec` 仍把包含 stack/guard 的 `sz` 回写到 `heap_end`，和模块一 `heap_end + vmas[] + sz` 的语义冲突

位置：`kernel/exec.c:130-138`，`kernel/proc.c:306-327`，`kernel/sysproc.c:92-110`，`kernel/vm.c:429-445`

这一条是更偏底层的风险，但它和你问的“shell 等基础设施还能怎么改、启动后部分命令会不会异常”有直接关系。

模块一文档已经把地址空间模型定义成：

1. `heap_end` 表示堆的真实边界。
2. `vmas[]` 表示独立映射区域。
3. `sz` 表示地址空间上界。

但当前 `exec()` 在提交新镜像时，仍然直接：

```c
p->sz = sz;
p->heap_end = sz;
```

而这里的 `sz` 已经包含了用户栈和 guard page 之后的总上界。这样一来，`heap_end` 就不再是“真实堆边界”，而变成了“整个用户空间当前顶端”。后续 `sbrk()`、lazy fault 和地址空间判断都会建立在错误口径上。

这条问题不一定让每个命令都立刻炸掉，但它会制造一种非常糟糕的状态：系统看起来能跑，但某些命令、某些分配路径、某些边界访问会出现带偶然性的异常。对于毕设演示来说，这类“偶发不稳”比“稳定报错”更难解释。

建议：

1. `exec()` 里单独保存“程序镜像结束位置”，用它设置 `heap_end`。
2. `sz` 仍然保留为包含用户栈后的地址空间上界。
3. 把 `echo hi`、`ps`、`mmap_test`、`scheduler_test` 之外，再补一个明确触发 `sbrk/malloc` 的启动后回归。

#### P2：前端布局现在总体合理，但调度页仍是“代理观察面板”，还没有完全达到模块一展示要求

位置：`dashboard/index.html:11-66`，`dashboard/js/main.js:1-171`，`dashboard/bridge/parser.py:388-434`，`kernel/sched/sched_mlfq.c:18-105`，`kernel/sched/sched_srtf.c:16-115`，`kernel/sched/sched_cfs.c:15-141`

这条我给的是 P2，不是因为它不重要，而是因为它已经从“错误”进入“下一轮增强”阶段了。

前端布局这轮其实是合理的：

1. 左侧导航 + 右侧工作区，符合任务管理器式观察路径。
2. 默认落地到 `#/processes`，比强制先看首页导览更适合教学演示。
3. 各页面独立轮询，滚动位置也做了保留，说明交互细节已经在收口。

但从“是否达到你的毕设题目”和“是否真正服务模块一机制展示”来看，它现在还有一个明显短板：**很多策略页用的还是代理指标，不是策略本体指标。**

当前 parser 自己已经承认：

1. `MLFQ` 还没有 `queue_level`。
2. `SRTF` 还没有 `remaining_time`。
3. `CFS` 还没有 `vruntime`。

所以调度页虽然已经有壳子，但对这三类算法的说明还停留在：

1. `MLFQ` 用 `retime/stime` 猜分层行为。
2. `SRTF` 用 `rutime/retime` 代替剩余时间。
3. `CFS` 用 `rutime` 分布近似公平性。

这能做演示，但还不够支撑“模块四把模块一讲活了”的强表述。更准确的说法是：**当前前端布局已经合格，当前调度数据面还没完全合格。**

建议：

1. 把 `queue_level`、`remaining_time`、`vruntime` 真正纳入快照接口。
2. 让 `Scheduler` 页优先展示“机制字段”，而不是一直停留在代理统计。
3. `Memory` 页继续保留 `sz != RSS` 的口径说明，这点当前方向是对的。

### 对架构、前端、论文主线和文档的判断

#### 1. Web 架构现在是否符合要求

从代码结构看，当前模块四已经基本符合“应该有独立架构，而不是临时脚本拼出来”的要求：

1. `dashboard/index.html` 只保留壳层。
2. `dashboard/js/` 已经拆成 `main/router/store/api/utils/views`。
3. host 侧 bridge 也拆成了 `app/state/parser`。

这个方向是对的，而且已经明显优于旧版单文件 `app.js`。

真正不符合要求的地方，不在“有没有拆文件”，而在“数据链是不是可信、字段语义是不是准确”。所以模块四下一步不应该再回头重做页面骨架，而应该把精力放在数据正确性和模块一语义对齐上。

#### 2. Web 前端布局是否合理

当前布局总体合理，可以继续沿着这个方向收口，不建议再退回大首页 + 纵向长滚动拼盘。

我对布局的评价是：

1. 桌面端方向正确，导航和工作区边界清楚。
2. 页面内卡片、表格和详情面板的组合方式也适合教学演示。
3. 还可以继续加强的是：每个页面都再多一块很短的“这个页面对应模块一里的哪条机制”说明，而不是回到大段首页文案。

也就是说，当前布局已经能用，下一步该补的是“机制映射”，不是大改视觉。

#### 3. 是否达到“扩展 xv6 教学操作系统”的毕设主题

当前模块四已经开始体现这个主题，但还差最后一层闭环：

1. 结构层面，它已经像一个教学运行态监控器了。
2. 数据层面，它还没有把模块一最关键的机制字段完整展示出来。
3. 可靠性层面，重复帧/丢帧和 `ps` 崩溃会削弱“研究平台”这个定位。

所以现在更准确的评价是：

> 模块四的产品骨架已经基本达到题目要求，但要真正成为“支撑实现、验证、实验、展示闭环的研究平台”，还必须先把数据链可靠性和字段语义修正到位。

#### 4. 文档质量

当前模块四文档的优点是方向清晰，边界也写得比较明确；问题是“历史问题”和“当前状态”已经开始混在同一份评审文档里。

例如这份 `docs/review/模块四.md` 前半段仍然保留了上一轮关于“单页拼盘”“stdout 默认链路”的历史审查结论，而后半段又记录了这些项已经整改完成。这样对后来读文档的人不够友好，需要反复比对日期才能知道什么是现状、什么是旧问题。

这次继续采用“追加新日期小节”的方式是对的。后续建议保持这个习惯，并且在文档顶部加一个很短的“当前有效结论索引”，避免评委或后续维护者读到过时段落后误判现状。

### 基础设施与 shell 的改进建议

这一部分我建议务实一点，不要把时间花在做一个“更像 Linux Bash 的 shell”上。对你这个毕设来说，更有价值的是把演示和回归链路做稳。

更值得投入的方向是：

1. 审计所有用户态工具，继续把大缓冲区从用户栈迁到静态区。
2. 给 `start_monitor.sh` 配一套最小 smoke 流程，至少覆盖：启动、`echo hi`、`ls`、`memstat`、`ps`、页面 API 在线。
3. 如果要增强 shell，优先做非常轻量的演示辅助能力，例如：
   - 一个切策略的小工具；
   - 一个统一跑演示命令的小工具；
   - 一个能快速输出“当前策略/关键指标”的 helper。
4. 不建议把大量时间放在 history、补全、复杂 quoting 这类和论文主线关系不大的 shell 功能上。

### 本轮结论

本轮复审后的结论可以压缩成一句话：

> 模块四现在的前端架构和页面布局已经基本走对了，但它当前最严重的问题已经从“没有架构”变成了“数据不够可信、字段不够准确、基础工具还不够稳”。

5. 最后再补 `queue_level`、`remaining_time`、`vruntime`，把调度页从“代理观察”升级成“机制观察”。

# 模块三架构设计

## 使用说明

本文档同时服务于两类场景：

1. 论文写作：可直接作为模块三“实验设计与实验方法”章节的基础材料，重点阅读“模块定位”“模块三到底要回答什么问题”“实验场景设计”“结果表达”和“最终结论”等部分。
2. 工程使用：如果你是要在仓库里实际复采数据、校验日志和重生成图表，建议重点阅读“数据链路架构”“正式数据集与论文口径”和“审查与整改后形成的最终口径”几节。

模块三当前最常用的实际入口是以下几步：

```bash
bash experiments/collect/collect_data.sh all --rounds 3 --resume
python3 experiments/analyze/validate_logs.py
python3 experiments/analyze/plot_results.py
```

如果目标是论文正式数据，请优先围绕 `COW + Lazy` 和当前保留的调度场景组织实验，不要再把 Buddy 当成 formal comparison 主线。

## 1. 模块定位

模块三真正要解决的问题，不是“怎么自动跑实验”，而是“在 xv6 + QEMU 这样的教学系统约束下，到底应该设计什么实验，才能让模块一里的机制差异被看见、被解释、被写进论文”。

如果没有模块三，模块一和模块二最多只能说明两件事：

1. 某些机制已经实现。
2. 某些关键路径已经通过回归验证。

但论文还需要第三层证据，也就是“这些机制在什么场景里有效、为什么有效、什么时候反而无效”。模块三承担的就是这一层证据组织工作。

因此，模块三在整个系统中的定位，不是简单的数据采集脚本集合，而是把“实验问题设计、结构化采集、质量门禁、结果表达”连成闭环的实验方法模块。

需要明确的是，当前论文主线中的正式内存实验只保留 `COW + Lazy`。Buddy 与 `mmap` 的实现仍保留在系统中，但它们不再承担 formal thesis memory comparison 的主证据角色。这不是因为功能不存在，而是因为当前没有形成足够稳、足够清楚、足够能支撑论文结论的正式实验问题。

## 2. 为什么要做这个模块

教学操作系统里的实验，很容易滑向两个误区。

第一，只要能自动采集和自动出图，就误以为实验已经成立。实际上，如果实验问题本身没有问准，那么后面采集、校验、画图做得再完整，也只是在把一批无效数字整理得更整齐。

第二，只要结果看起来“有提升”，就急于把它写成机制优势。但模块一中的调度、COW、Lazy Allocation 和 `mmap` 都不是无条件优化，它们的收益高度依赖场景。如果实验没有把场景设计清楚，就很容易把条件性结论误写成普遍结论。

因此，模块三要解决的核心问题不是“怎么跑实验”，而是下面几个更本质的问题：

1. 模块一里哪些机制值得进入正式性能比较，哪些只适合作为实现和验证项。
2. 每组实验到底在回答什么问题，而不是笼统地问“性能有没有提升”。
3. 如何把实验日志变成结构化、可追溯、可审计的数据链路。
4. 如何在正式结论生成前，阻断无效数据和混口径数据进入论文。

从这个角度看，模块三的真正价值不在于脚本本身，而在于它把“实验问题”放在了“自动化流程”之前。

## 3. 模块目标

模块三的设计目标可以概括为以下四点：

1. 为模块一中的关键机制设计问题明确、语义清楚的实验场景。
2. 建立结构化的数据采集、清洗、校验、聚合与出图链路。
3. 为论文提供可复采、可审计、可解释的正式数据集。
4. 将“实现已存在”和“结论可成立”之间补上一条可靠的方法学桥梁。

## 4. 模块三到底要回答什么问题

在当前论文口径下，模块三围绕三个具体问题展开：

1. `COW` 到底在什么访问模式下能显著降低 `fork()` 成本，什么情况下反而会吃亏。
2. `Lazy Allocation` 到底在什么访问密度下有效，什么时候只是把分配成本从 `sbrk()` 挪到了缺页异常。
3. 不同调度策略在教学 OS 的单核 tick 环境下，是否能在可控 workload 中表现出可解释的差异。

这三个问题都不是抽象的“性能有没有提升”，而是“某种机制为什么在某类场景里有效或无效”。这也是模块三必须以场景设计为中心，而不能只以自动化流程为中心的原因。

## 5. 为什么正式内存实验只保留 COW 和 Lazy

### 5.1 COW 为什么值得保留

`COW` 的价值不在于“任何情况下都更快”，而在于它把复制时机从 `fork()` 时刻推迟到真正发生写入时。因此，评价 COW 的关键不是一个统一均值，而是写入比例如何改变收益。

这使得 COW 非常适合设计成一组有明确机制指向的正式实验。

### 5.2 Lazy 为什么值得保留

`Lazy Allocation` 的核心也不是“分配更快”，而是“只为真正访问的页付出物理分配成本”。因此，评价 Lazy 是否有效，关键不在申请了多大地址空间，而在申请后到底访问了多少。

这种“访问密度驱动收益”的特征，同样很适合被设计成一组场景明确、边界清楚的正式实验。

### 5.3 为什么 Buddy 和 mmap 不再作为正式比较项

这是一次主动收敛，而不是被动放弃。

1. Buddy 当前没有形成像 `COW + Lazy` 那样语义清楚、正反例明确、能直接支撑论文主结论的正式比较问题。
2. `mmap` 当前更适合写成“已实现并完成验证的机制扩展”，而不是 baseline/testing 对称性能比较的正式证据。

论文里最忌讳的，不是“少做一张图”，而是“用一张语义不稳的图去支撑一个很重的结论”。因此，模块三当前主动缩小正式内存结论范围，是为了保证结论可信，而不是因为自动化能力不够。

## 6. 实验场景设计：为什么这样拆问题

### 6.1 COW 场景设计

当前保留的四个场景分别对应四个问题：

1. `COW_NO_ACCESS`
目的：验证子进程 `fork()` 后几乎不使用地址空间时，COW 能否避免无意义整页复制。

2. `COW_READONLY`
目的：验证只读共享是否能够稳定保留 COW 的收益，而不是被额外机制开销抵消。

3. `COW_PARTIAL`
目的：验证只写一部分页面这种最接近真实使用模式的中间情形，观察收益是否仍然存在。

4. `COW_FULLWRITE`
目的：给出明确反例，证明 COW 不是无条件更优；当子进程最终写满全部页面时，缺页复制会把延迟成本重新付回来。

因此，COW 这组实验真正支持的结论应当是：COW 的收益依赖于写入比例，而不是对所有 `fork()` 场景都更优。

### 6.2 Lazy 场景设计

当前保留的三个场景分别对应三种访问密度：

1. `LAZY_SPARSE`
目的：模拟“申请得多、实际只碰很少页面”的稀疏使用模式。这是 Lazy 的最佳场景。

2. `LAZY_HALF`
目的：观察访问密度上升到中等水平后，Lazy 的收益是否开始缩小，甚至出现拐点。

3. `LAZY_FULL`
目的：给出明确反例。当页面最终全部被访问时，Lazy 只是把成本分散到了多个缺页点，不应再被包装成优势。

因此，Lazy 这组实验真正回答的是：访问密度如何影响 on-demand 分配的收益，而不是 Lazy 是否永远优于 eager allocation。

### 6.3 调度场景设计

调度算法的比较也不能只靠一个总分，因为不同策略优化的目标本来就不同。模块三当前保留四类场景，每类场景只问一个问题：

1. `throughput`
实际测的是混合批任务的平均完成时间，用来看不同策略处理长短任务混合时的整体完成效率。

2. `convoy`
实际测的是一个长任务在前、多个短任务在后的情况下，短任务平均要等多久，目的是观察 convoy effect 是否被放大或缓解。

3. `fairness`
实际测的是等工作量任务之间的完成时间离散度，用来近似观察不同策略是否会让某些任务明显“多等很多”。

4. `response`
实际测的是后台 CPU hog 已经存在时，新到短任务的平均完成延迟，用来近似观察策略对“后到小任务”的照顾程度。

这里每个场景都不是为了抽象证明某算法“理论上最优”，而是为了在 xv6/QEMU 这种单核、tick 粒度较粗的环境里，用尽量少的变量问出一个明确问题。

## 7. 为什么模块三不能只靠“自动跑一堆程序”

教学 OS 的实验和工业系统 benchmark 最大的差别在于：环境更小、噪声更大、分辨率更低。如果场景设计不够有针对性，就会出现两类假象：

1. 算法其实有差别，但 workload 太轻，最后全部塌到 `1-3 ticks`。
2. 机制其实只在部分场景有效，但因为只测了一个点，最后被误写成“全面优化”。

所以，模块三的方法论必须先是“问题设计”，再是“数据链路”。自动化是保证实验整理不失真的手段，不是实验本体。

## 8. 数据链路架构：从 perftest 到论文产物的闭环

当前实验链路为：

```text
user/perftest.c
-> experiments/collect/collect_data.sh
-> experiments/analyze/experiment_data.py
-> experiments/analyze/validate_logs.py
-> experiments/analyze/plot_results.py
```

这条链路是模块三最重要的工程成果，因为它把“实验场景”真正落成了“可反复执行、可追溯审计、可生成论文产物”的数据闭环。

### 8.1 `perftest.c`：结构化实验输出生产者

[ user/perftest.c ](/home/niya/xv6/user/perftest.c) 的作用，不只是跑测试，而是把实验场景转化为结构化文本输出。它会输出三类记录：

- `META:ts=<ticks>:cpus=<n>:branch=<name>:sched=<name>`
- `RESULT:<name>:<avg>:<std>:<unit>:<batch>:<runs>`
- `SAMPLES:<name>:<count>:<values>`

这一步很关键，因为模块三后续的采集、校验、聚合和出图都依赖这一输出契约。如果没有统一结构化输出，后面的数据链路就只能依赖脆弱的日志切割和人工解释。

### 8.2 `collect_data.sh`：多轮采集与审计入口

[collect_data.sh](/home/niya/xv6/experiments/collect/collect_data.sh) 负责真正执行多轮采集。它的作用包括：

1. 支持 baseline / testing 两个分支的独立采集。
2. 支持多轮 round-level 运行，并保留 resume 能力。
3. 支持把默认 `perftest all` 展开为 split boot 形式，避免把 memory 和 scheduler 结果混在同一次启动中。
4. 为每一轮运行写出 `manifest.tsv`，记录 commit、命令、CPU 数、timeout、完成状态、失败命令等元信息。
5. 同时保留 raw log 和 clean log，为后续审计与恢复提供依据。

也就是说，`collect_data.sh` 的核心不是“自动化运行”，而是把每一轮实验变成可追溯、可审计的数据单元。

### 8.3 `experiment_data.py`：统一数据模型与聚合层

[experiment_data.py](/home/niya/xv6/experiments/analyze/experiment_data.py) 是模块三的数据中台。它负责：

1. 解析 `META`、`RESULT`、`SAMPLES` 三类结构化输出。
2. 定义正式数据集中哪些测试是必需项。
3. 定义 `throughput`、`convoy`、`fairness`、`response` 等场景分组。
4. 按 run、按 branch、按场景聚合数据。
5. 维护 `TEST_META`，把原始测试名映射成更适合论文写作和图表呈现的标签。

这一层的重要意义在于：它把原始日志提升成统一数据模型，使模块三后续脚本不再直接操作杂乱文本，而是围绕统一对象工作。

### 8.4 `validate_logs.py`：正式结论前的质量门禁

[validate_logs.py](/home/niya/xv6/experiments/analyze/validate_logs.py) 的角色不是简单“检查文件存在”，而是做正式结论前的质量门禁。它会检查：

1. 是否存在缺失的必需测试项。
2. 是否存在混合 signature、混合 command、混合 commit 等口径问题。
3. 是否存在 partial run、raw-only run、manifest 不一致等可疑情况。
4. 是否存在低分辨率、低轮数、高波动等会削弱论文说服力的问题。
5. testing 分支是否采用 split boots，避免调度实验被前面的 memory workload 污染。

因此，模块三不是“采完就画”，而是必须先经过质量门禁。只有通过门禁的数据，才配进入正式报告和论文结论。

### 8.5 `plot_results.py`：从聚合数据到论文产物

[plot_results.py](/home/niya/xv6/experiments/analyze/plot_results.py) 负责把已经通过门禁的数据生成正式产物，包括：

1. 图像文件，如 `.png` / `.pdf`
2. 数据汇总文件，如 `.json` / `.csv`
3. Markdown 报告
4. LaTeX 表格，如 `thesis_memory_table.tex`

这一步的意义在于，它把实验结论从“内部数据”转换为“可以直接进入论文和答辩材料的表达形式”。

## 9. 正式数据集与论文口径

当前正式数据集统一使用：

- `experiments/logs/thesis_final/baseline`
- `experiments/logs/thesis_final/testing`

在论文中，更准确的表述应当是：

1. 模块三构建了一个可复采、可审计的实验链路。
2. 该链路当前正式支撑的内存结论只覆盖 `COW + Lazy`。
3. 这些实验说明的不是“所有优化都全面提升”，而是“某类机制在某类访问模式下为什么有效、为什么无效”。
4. 调度实验结论应写成“在当前 xv6/QEMU workload 下观察到的趋势”，而不是普遍定律。

## 10. 结果表达：模块三当前到底证明了什么

### 10.1 COW 证明的不是“全面更快”，而是“收益取决于写入比例”

1. 在 `NO_ACCESS` 与 `READONLY` 场景中，COW 收益显著。
2. 在 `PARTIAL` 场景中，收益仍然存在，但明显收缩。
3. 在 `FULLWRITE` 场景中，COW 成为明确反例。

因此，COW 这组实验真正支持的写法应是：COW 对低写入比例的 `fork()` 使用模式显著有利，但并非对所有进程行为都更优。

### 10.2 Lazy 证明的不是“分配更快”，而是“收益取决于访问密度”

1. 在 `SPARSE` 场景中，Lazy 收益最明显。
2. 在 `HALF` 场景中，Lazy 已经不再稳定占优。
3. 在 `FULL` 场景中，Lazy 成为明确反例。

因此，Lazy 这组实验真正支持的写法应是：Lazy 更适合稀疏访问的地址空间，而不是总是优于 eager allocation。

### 10.3 调度实验支持的是“趋势”，不是“普遍定律”

当前调度实验适合支撑“在当前 workload 下观察到的趋势”，而不是抽象成普遍定律。例如：

1. 某些混合批任务场景下，RR 当前表现最好。
2. convoy 场景下，RR 更能保护短任务完成时间。
3. fairness 场景下，RR 与 CFS 接近，而 PRIORITY 明显更差。
4. response 场景下，MLFQ 对后到短任务更友好。

这些结论可以写，但必须始终带着场景限定语。

## 11. 关键设计取舍

### 11.1 为什么强调“一个实验先回答一个问题”

如果一个实验同时想证明太多东西，最后往往什么都证明不清。模块三当前的一个核心原则就是：一组实验先只回答一个问题。只有这样，结果中的正例和反例才有明确解释空间。

### 11.2 为什么正式结论只保留语义最稳的部分

不是所有实现过的机制都适合进入论文主结论。模块三当前主动只保留 `COW + Lazy` 作为正式内存实验，就是因为它们满足三个条件：

1. 为什么设计这些场景是清楚的。
2. 场景与机制之间的对应关系是清楚的。
3. 结果中既有正例，也有反例，足以支撑边界化结论。

### 11.3 为什么自动化链路服务于问题，而不是替代问题

`collect_data`、`validate_logs`、`plot_results` 这些工具仍然非常重要，但它们的角色是：

1. 保证数据来源可追溯。
2. 保证无效数据不会混进正式结论。
3. 保证图表和表格能稳定重生成。

它们不能替代实验设计本身。模块三真正困难的部分，始终还是“应该测什么、为什么这样测”。

## 12. 审查与整改后形成的最终口径

结合现有实验数据和整改结果，模块三当前可以给出如下更准确的判断：

1. 模块三真正有价值的地方，不是“自动生成了多少图”，而是把问题设计、结构化采集、质量门禁和结果表达连成了闭环。
2. 当前最适合写进论文正文的正式内存实验是 `COW + Lazy`。
3. Buddy 与 `mmap` 仍可作为实现层扩展或附录材料，但不再写成论文主证据。
4. 调度实验当前支持的是在指定 xv6/QEMU workload 下观察到的趋势，而不是脱离场景的普遍定律。

### 12.1 当前整改后形成的原则

1. 一个实验先回答一个问题。
2. 正式结论只保留语义最稳的部分。
3. 自动化链路服务于问题，而不替代问题。

### 12.2 仍需诚实保留的边界

1. formal dataset 可信，不等于每个指标都低噪声。
2. 高波动 warning 仍然存在，它们提醒的是解释边界，而不是链路失效。
3. 正式内存性能结论当前只收敛到 `COW + Lazy`。
4. 调度实验应始终保留 workload 限定语。

## 13. 验证闭环：模块三如何服务整篇论文

模块三在整篇论文中的闭环关系可以概括为：

模块一提出机制与系统设计 -> 模块二提供正确性回归门禁 -> 模块三围绕清楚的问题设计正式实验 -> 结构化采集并通过质量门禁 -> 输出图、表、报告和正式结论 -> 模块四进一步承担可视化展示与交互表达。

这意味着模块三不是“补几张图”，而是把论文中的“实验设计”和“实验结果”真正落到了可复查的方法学链路上。

## 14. 最终结论

模块三当前已经可以支撑论文结论，但支撑的是“问题问得准的那一部分”。

从架构角度看，它完成了三件关键事情：

1. 把机制比较转化成问题明确、场景清楚的正式实验设计。
2. 建立了从 `perftest` 到报告、图表和表格的结构化数据链路。
3. 通过 `manifest + validation + plotting` 把实验结果变成可复采、可审计、可表达的论文证据。

因此，更准确的论文表述应当是：模块三通过问题导向的实验设计和可追溯的数据链路，为模块一中的关键机制提供了正式实验支撑；其中当前最稳、最适合进入论文正文的内存结论集中在 `COW + Lazy`，而不是把所有已实现功能都等量写成正式对比结论。

# 模块四总览与方案

模块四承担的是本项目的展示与消费层职责。它的目标不是重新定义底层数据协议，而是把模块一到模块三已经沉淀出的结构化状态与正式实验产物组织成更直观的观察入口。从毕设闭环的角度看，模块四不是新的研究对象，而是把前面三个模块的成果转化为“可展示、可答辩、可维护”的系统界面。

## 设计动机

如果展示层直接消费 xv6 原始控制台输出，就会面临两个问题。第一，控制台输出格式天然偏向调试，不适合作为稳定接口；第二，一旦模块一或模块三调整输出内容，前端就需要被动追着改，导致展示层与内核实验层高度耦合。与此同时，本项目已经在模块三中形成了 `results.json`、CSV、Markdown 报告和 LaTeX 表格等标准化产物，因此再为展示层重新发明一套协议，不但收益有限，反而会增加系统复杂度。

因此，模块四的设计原则是：展示层优先复用既有结构化产物，只在确有必要的地方补充系统快照链路，而不反向定义新的底层数据规范。

## 模块目标

1. 为系统关键状态和实验结果提供更直观的消费入口。
2. 复用模块三已有的结构化产物，避免重复解析原始日志。
3. 为答辩展示和后续维护提供轻量、稳定的数据界面。
4. 在 xv6 小栈、QEMU 单机环境约束下，采用风险更低的监控与展示方案。

## 原理设计

模块四采用双数据入口设计：

1. 静态实验结果入口
2. 动态系统快照入口

### 静态实验结果入口

静态实验结果主要来自模块三已经生成的标准化产物，包括：

- `experiments/outputs/data/results.json`
- `experiments/outputs/data/comparison_summary.csv`
- `experiments/outputs/data/scheduler_summary.csv`
- `experiments/outputs/reports/experiment_report.md`
- `experiments/outputs/figures/*`

这部分数据适合支撑“结果展示”类页面，例如总体概览、内存优化对比、调度场景比较和论文图表回放。由于这些产物已经经过模块三的校验与聚合，因此展示层无需再承担数据质量判断职责。

### 动态系统快照入口

动态状态观测通过 `user/sysmon.c` 实现。`sysmon` 不直接输出面向人类排版的表格，而是输出结构化 `SNAP` 与 `PROC` 记录：

1. `SNAP` 记录系统级快照，例如当前调度器、上下文切换次数、总页数、空闲页数、运行队列长度、Buddy/COW/Lazy 计数等。
2. `PROC` 记录进程级快照，例如 PID、状态、优先级、票数、调度类、内存大小和运行统计。

随后，宿主机侧的 `dashboard/parse_sysmon.py` 将这些记录解析为 `dashboard/data/snapshots.json`，供前端页面消费。这样的设计将“内核状态导出”和“前端展示格式”明确拆分，降低了展示层改动对内核输出协议的影响。

## 核心数据结构、接口与流程

### 数据流一：实验结果展示链路

```text
experiments/logs/thesis_final
-> experiment_data.py / plot_results.py
-> experiments/outputs/data/*.json/*.csv
-> dashboard/index.html + dashboard/js/app.js
```

这一链路的关键特点是：前端读取的是已经过验证和聚合的静态结果，而不是未清洗的原始日志。这样可以保证页面展示与论文图表使用同一数据口径。

### 数据流二：监控快照链路

```text
user/sysmon
-> SNAP / PROC structured text
-> dashboard/parse_sysmon.py
-> dashboard/data/snapshots.json
-> dashboard/index.html + dashboard/js/app.js
```

这一链路的关键特点是：用户态监控程序只负责导出结构化状态，宿主机脚本负责转换，前端只负责消费 JSON。三层职责清晰，便于逐层排错。

### 当前页面职责划分

按照当前方案，模块四至少应覆盖以下四类页面或视图：

1. 概览页：展示项目状态、当前调度器、最近一次正式实验摘要。
2. 内存页：展示 COW、Lazy、Buddy、`mmap` 的关键指标与反例。
3. 调度页：展示 throughput、convoy、fairness、response 四类场景比较。
4. 监控页：展示系统快照与进程快照的实时或准实时结果。

## 关键设计取舍

### 为什么优先复用模块三产物，而不是重新解析原始日志

因为模块三已经承担了“解析、聚合、校验和解释边界”的职责。若模块四再直接读取原始日志，就会把数据清洗逻辑重新搬一遍，既容易造成口径不一致，也会让前端承担不必要的复杂性。复用结构化产物能够保证：论文、命令行报告和 Web 页面看到的是同一套结果。

### 为什么监控链路采用“结构化文本 -> JSON”的两步法

xv6 用户态程序输出能力有限，宿主机脚本处理能力更强。若要求 `sysmon` 直接生成复杂 JSON，不仅会增加用户态工具复杂度，还会放大栈空间和字符串处理的风险。当前方案让 `sysmon` 只输出稳定的键值对记录，再由宿主机脚本负责 JSON 化，是更适合教学系统约束的折中设计。

### 为什么模块四当前应写成“方案明确、工程待推进”

从整个毕设闭环看，模块四的重要性在于“消费已有成果”，而不是继续扩展底层系统能力。当前项目已经具备监控导出工具和结构化实验结果，但页面编排、交互体验和工程细节仍未完全收口。因此，模块四最准确的状态应是“设计方向和数据入口已经明确，基础链路打通，展示层工程化仍有空间”，而不是“完整 Web 平台已全部完成”。

## bug 如何推动方案收敛

### `BUG-018`：用户态监控工具未覆盖新调度器

该问题说明展示层与观测层必须随内核能力同步演进。若调度器已经扩展到 9 种策略，而监控工具仍按旧枚举输出名称，那么展示结果就会与真实系统状态不一致。修复后，模块四更强调“观测模型必须跟随内核接口同步更新”。

### `BUG-020`：`sysmon` 栈溢出导致监控卡住

该问题直接体现了 xv6 用户态环境的资源约束。`sysmon` 早期把大缓冲区放在栈上，导致程序在真正进入采样循环前就破坏了自身状态。修复后，大缓冲区被迁移到静态存储区，并在代码中明确保留“小栈约束”说明。这一问题使得模块四不再追求复杂重型监控程序，而是倾向于稳定、轻量、协议简单的工具设计。

## 验证方式与当前状态

模块四当前的可验证成果主要包括：

1. `sysmon` 已能稳定输出结构化 `SNAP/PROC` 记录。
2. `parse_sysmon.py` 已能将原始记录转换为 `snapshots.json`。
3. Web 层已有 `dashboard/index.html`、`dashboard/js/app.js` 与快照数据入口。
4. 模块三产物已经可以直接被展示层复用。

因此，模块四并非“从零开始”，但也尚未进入完全收尾状态。更准确地说，它已经完成了“数据入口与总体方案”的闭环，剩余工作主要集中在展示层工程细化，而不再是底层观测协议设计。

## 已知边界与论文表述建议

1. 模块四当前不宜写成完整实时监控平台，而应写成“展示层方案与基础链路已通”。
2. 页面层的视觉与交互并不是本项目的核心研究贡献，论文中应重点强调其对前面三模块成果的消费关系。
3. 动态快照链路的目标是提供状态观察能力，而非替代正式实验数据。
4. 论文中应说明模块四优先消费模块三正式产物，这一点体现了整个项目的系统化组织能力。

# 模块一：调度算法优化架构设计（对齐当前实现）

## 一、文档说明

本文档用于对齐当前仓库中的模块一调度器实现，描述对象为当前 `testing` 分支上已经提交并通过最小回归验证的版本。文档重点不再是“理想算法介绍”，而是“当前 xv6-riscv 代码里真实采用的架构设计、接线方式、数据流和边界约束”。

## 二、设计背景与目标

原始 xv6 只有一个简单的 Round-Robin 调度器，便于教学，但不足以支撑“多种调度策略对比实验”和“调度器框架化设计”这两类本科毕设目标。因此本模块的目标不是去复刻 Linux，而是在 xv6/QEMU 的约束下，实现一个：
1. 能运行时切换策略的调度框架。
2. 能容纳多种经典算法的统一接口。
3. 能在不破坏 xv6 原始调度主循环的前提下，给不同算法接入自己的状态更新逻辑。
4. 能为模块二测试、模块三实验和模块四监控提供统一的统计与状态读取入口。

本模块当前实现的 9 种调度策略如下：

| 策略 | 枚举值 | 类型 | 当前实现定位 |
|------|--------|------|--------------|
| DEFAULT | 0 | Round-Robin | xv6 基线调度器 |
| FCFS | 1 | 非抢占 | 批处理对比基线 |
| PRIORITY | 2 | 优先级 | 基于 `priority` 的简单优先级调度 |
| SML | 3 | 静态多级队列 | 基于 `sched_class` 的三级静态队列 |
| LOTTERY | 4 | 概率调度 | 基于 `tickets` 的彩票调度 |
| SJF | 5 | 非抢占 | 基于历史运行时间估计的简化 SJF |
| SRTF | 6 | 抢占式 | 基于剩余时间递减的简化 SRTF |
| MLFQ | 7 | 反馈队列 | 具备时间片降级与唤醒提升的简化 MLFQ |
| CFS | 8 | 公平调度 | 基于 `vruntime` 的简化 CFS |

## 三、总体架构

### 3.1 架构分层

当前调度模块的总体结构如下：

```text
模块一调度子系统
├── 调度框架层
│   ├── kernel/sched/sched.h
│   └── kernel/sched/sched.c
├── 策略实现层
│   ├── sched_default.c
│   ├── sched_fcfs.c
│   ├── sched_priority.c
│   ├── sched_sml.c
│   ├── sched_lottery.c
│   ├── sched_sjf.c
│   ├── sched_srtf.c
│   ├── sched_mlfq.c
│   └── sched_cfs.c
├── 生命周期接线层
│   ├── kernel/proc.c      进程创建、退出、唤醒
│   ├── kernel/trap.c      时钟中断触发调度统计
│   └── kernel/sched/sched_stats.c
└── 用户接口层
    └── kernel/sysproc.c   setscheduler / getscheduler / setpriority / getstats
```

### 3.2 设计核心思想

这套架构的关键不是“把 9 个算法都写出来”，而是把“策略选择”和“策略状态维护”解耦：

- 调度主循环仍由 xv6 原生 `scheduler()` 驱动。
- 真正的“挑选下一个进程”逻辑由当前策略的 `pick_next()` 决定。
- 真正的“算法私有状态更新”不再散落在各个策略文件里自说自话，而是通过统一生命周期钩子接到内核关键路径上。

因此，调度框架采用的是“策略接口 + 生命周期钩子”的结构，而不是单纯的函数表切换。

## 四、核心数据结构

### 4.1 调度器接口 `struct sched_ops`

当前接口定义于 `kernel/sched/sched.h`：

```c
struct sched_ops {
    const char *name;
    void (*proc_init)(struct proc *p);
    struct proc* (*pick_next)(void);
    void (*proc_exit)(struct proc *p);
    void (*proc_tick)(struct proc *p);
    void (*proc_wakeup)(struct proc *p);
};
```

各字段语义如下：

- `name`：调度器名称，用于调试、监控和用户态显示。
- `proc_init`：策略私有状态初始化。`p == 0` 时表示初始化整套策略状态；`p != 0` 时表示初始化某个具体进程。
- `pick_next`：从 `proc[]` 中选择一个 `RUNNABLE` 进程并执行上下文切换。
- `proc_exit`：进程退出时更新策略状态，例如 SJF/SRTF 的估计时间、CFS 的 `vruntime` 回收。
- `proc_tick`：时钟 tick 到来时更新当前运行进程的算法私有状态，例如 SRTF 的 `remaining_time`、MLFQ 的队列用时、CFS 的 `vruntime`。
- `proc_wakeup`：进程从睡眠态被唤醒时更新其调度优先级，用于 MLFQ 这类需要把交互型进程往高优先级队列移动的策略。

### 4.2 调度策略枚举

```c
enum sched_policy {
    SCHED_DEFAULT,
    SCHED_FCFS,
    SCHED_PRIORITY,
    SCHED_SML,
    SCHED_LOTTERY,
    SCHED_SJF,
    SCHED_SRTF,
    SCHED_MLFQ,
    SCHED_CFS,
};
```

该枚举一方面作为 `setscheduler()` 的用户态参数，另一方面作为 `sched_ops_table[]` 的索引。

### 4.3 `struct proc` 中与调度相关的字段

当前 `struct proc` 中保留了统一的、跨策略共享的调度字段：

```c
int priority;
int tickets;
uint64 ctime;
int rutime;
int retime;
int stime;
int sched_class;
```

这些字段的职责分别是：

- `priority`：PRIORITY 调度器使用，范围为 `1..20`。
- `tickets`：LOTTERY 调度器使用。
- `ctime`：进程创建时间，常用于 FCFS 和 tie-break。
- `rutime`：累计运行时间，供统计、实验和部分算法使用。
- `retime`：就绪态等待时间。
- `stime`：睡眠时间。
- `sched_class`：SML 使用的静态队列类别。

另外，SJF/SRTF/MLFQ/CFS 这类策略需要的私有数组并没有直接塞进 `struct proc`，而是放在各自的 `sched_xxx.c` 中用 `NPROC` 大小的并行数组维护。这么做的原因是：

- 避免把每个算法的私有状态污染到所有进程结构中。
- 保持新增算法时的改动面集中在调度子目录。
- 便于在 `proc_init(0)` 中一次性整体初始化。

## 五、调度框架设计

### 5.1 调度器注册

`sched.c` 中维护了一个调度器注册表：

```c
static struct sched_ops* sched_ops_table[SCHED_CFS + 1] = {
    [SCHED_DEFAULT]  = &default_ops,
    [SCHED_FCFS]     = &fcfs_ops,
    [SCHED_PRIORITY] = &priority_ops,
    [SCHED_SML]      = &sml_ops,
    [SCHED_LOTTERY]  = &lottery_ops,
    [SCHED_SJF]      = &sjf_ops,
    [SCHED_SRTF]     = &srtf_ops,
    [SCHED_MLFQ]     = &mlfq_ops,
    [SCHED_CFS]      = &cfs_ops,
};
```

这使得“新增一个调度器”的流程非常清晰：

1. 新建 `kernel/sched/sched_xxx.c`。
2. 导出一个 `struct sched_ops xxx_ops`。
3. 在 `sched.h` 增加枚举值。
4. 在 `sched.c` 注册到 `sched_ops_table[]`。

### 5.2 当前策略状态

调度框架维护两个全局变量：

- `current_policy`：当前使用的调度策略编号。
- `current_ops`：当前策略对应的接口表。

再配合 `sched_lock`，实现策略切换时的基本保护。

### 5.3 启动初始化

内核启动阶段的初始化路径如下：

1. `main()` 调用 `sched_init()`。
2. `sched_init()` 初始化 `sched_lock`。
3. 默认策略设为 `SCHED_DEFAULT`。
4. `sched_init_policy_state(current_ops)` 被调用：
   - 先执行一次 `ops->proc_init(0)`，清空该策略自己的并行数组。
   - 再扫描整个 `proc[]`，对所有非 `UNUSED` 槽位补做 `ops->proc_init(p)`。

这一步非常重要，因为它解决了“切换策略之后旧槽位状态残留”的问题。

### 5.4 运行时切换策略

`sys_setscheduler()` 最终调用 `sched_set_policy()`，切换步骤如下：

1. 校验 `policy` 是否在有效范围内。
2. 从 `sched_ops_table[]` 取出新的策略对象。
3. 先调用 `sched_init_policy_state(new_ops)`，把新策略的全局状态和现有进程状态对齐。
4. 在 `sched_lock` 保护下更新 `current_policy` 和 `current_ops`。

这种设计意味着：

- 切换策略不是单纯地替换一个函数指针。
- 新策略会在接管之前主动为所有现存进程初始化自身状态。
- 对 SJF/SRTF/CFS 这种需要历史数组的策略尤其关键。

### 5.5 调度主循环保持 xv6 原型

`kernel/proc.c` 中的 `scheduler()` 没有被彻底推翻，而是保留了 xv6 原生结构：

1. CPU 进入死循环。
2. 开中断、关中断，避免错过设备唤醒。
3. 调用 `sched_pick_next()`。
4. 若返回空，则执行 `wfi` 等待中断。
5. 若返回非空，说明具体策略已经在 `pick_next()` 内部完成了上下文切换。

这样做的好处是：

- 改动面小，便于继承 xv6 原始调度语义。
- 所有策略都共享同一个调度壳，不需要各自复制一份 scheduler 主循环。
- 论文里可以清楚说明“保留 xv6 调度外壳，只替换策略选择器”。

## 六、生命周期钩子如何接入内核关键路径

这是当前实现最核心、也最值得在论文里展开的一部分。

### 6.1 进程创建：`allocproc()`

在 `kernel/proc.c` 中，进程槽位从 `UNUSED` 变成 `USED` 后，会完成以下初始化：

- 设置 `priority`、`tickets`、`ctime`、`rutime`、`retime`、`stime`、`sched_class`。
- 调用 `sched_proc_init_hook(p)`。

这一步把“新进程进入系统”事件交给当前策略，让 SJF/SRTF/MLFQ/CFS 能建立自己的私有状态。

### 6.2 进程退出：`kexit()`

进程退出时，在把状态切到 `ZOMBIE` 之前调用：

```c
sched_proc_exit_hook(p);
```

这一步负责把“运行后的结果”回灌给策略，例如：

- SJF/SRTF 根据 `p->rutime` 更新历史估计。
- CFS 重新维护 `min_vruntime`。
- MLFQ 清理对应槽位的队列级别和已用时间。

### 6.3 时钟 tick：`sched_update_stats()`

`sched_update_stats()` 位于 `kernel/sched/sched_stats.c`，由 `clockintr()` 每个时钟中断调用。它做两类事情：

1. 维护通用统计：`rutime`、`retime`、`stime`、total ticks、idle ticks、context switches 等。
2. 在确认当前进程处于 `RUNNING` 后调用：

```c
sched_proc_tick_hook(current);
```

因此，所有需要“每个 tick 更新”的算法都统一挂接到这个入口，而不是各自去修改 `trap.c`。

### 6.4 进程唤醒：`wakeup()`

在 `kernel/proc.c` 的 `wakeup()` 中，当一个睡眠进程被唤醒并转为 `RUNNABLE` 时，会调用：

```c
sched_proc_wakeup_hook(p);
```

这一钩子目前主要服务于 MLFQ，使交互型进程在 I/O 完成后能够向高优先级队列提升一级。

## 七、各调度算法实现说明

### 7.1 DEFAULT

- 定位：xv6 的默认 Round-Robin。
- 选择方式：顺序扫描 `proc[]`，遇到第一个 `RUNNABLE` 进程即运行。
- 作用：作为基线调度器，也是所有实验的对照组之一。

### 7.2 FCFS

- 依据：`ctime` 最小者优先。
- 类型：非抢占。
- 特点：实现简单，适合展示吞吐量与护航效应。

### 7.3 PRIORITY

- 依据：`priority` 值，数值越小优先级越高。
- tie-break：通常退化到创建时间或扫描顺序。
- 用途：展示“静态优先级”与交互性之间的取舍。

### 7.4 SML

- 依据：`sched_class`，静态三级队列。
- 特点：队列类别由用户或测试设置，运行时不会自动反馈调整。
- 定位：作为“静态分层调度”的教学型实现。

### 7.5 LOTTERY

- 依据：`tickets` 数量参与随机抽签。
- 特点：可以用于说明概率公平，而不是严格顺序公平。

### 7.6 SJF

SJF 是当前实现中的“简化版短作业优先”：

- 私有状态：`estimated_time[]`、`actual_time[]`、`burst_count[]`。
- 初始化：新进程默认估计时间为 `100`。
- 选择逻辑：在所有 `RUNNABLE` 进程中选择 `estimated_time` 最小者。
- 更新逻辑：在 `proc_exit` 中根据 `p->rutime` 更新估计值。

需要诚实说明的是：

- 当前 EWMA 更新使用的是累计运行时间风格的简化估计，不是严格按 CPU burst 分段统计。
- 因为 xv6 缺少复杂的调度队列和 burst 观测机制，这种实现更适合教学和对比实验，而不是宣称完全还原经典操作系统教科书中的 SJF。

### 7.7 SRTF

SRTF 在当前版本里已经形成闭环：

- 私有状态：`estimated_time[]`、`remaining_time[]`、`actual_time[]`、`burst_count[]`。
- 初始化：`remaining_time` 初始为默认估计值。
- 选择逻辑：优先选择 `remaining_time` 最小者。
- tie-break：若剩余时间相同，则选 `ctime` 更早的进程。
- tick 更新：`sched_proc_tick_hook()` 每个 tick 将当前运行进程的 `remaining_time` 减 1。
- 退出更新：进程结束时重新估计历史时间，并重置剩余时间。

因此，当前实现已经不是“伪 SRTF”，而是一个能够随运行时间递减剩余估计的简化抢占式调度器。

### 7.8 MLFQ

MLFQ 当前实现是“简化但可运行”的版本：

- 队列数：3 级。
- 时间片：`10 / 20 / 40` ticks。
- 私有状态：`rr_index[]`、`queue_level[]`、`time_in_queue[]`。
- 选择逻辑：总是先从高优先级队列扫描，再在队列内按 RR 轮转。
- tick 更新：当 `time_in_queue[idx]` 达到该队列时间片后，进程下降一级。
- wakeup 更新：从睡眠态被唤醒时，若不在最高级，则提升一级，并清零当前队列累计时间。
- 退出更新：清理该槽位的队列级别和时间计数。

需要说明的限制：

- 当前版本没有周期性全局 boost。
- 因此它可以表述为“具备时间片降级和交互唤醒提升的简化 MLFQ”，而不是 Linux/Unix 级别的完整反馈队列系统。

### 7.9 CFS

CFS 当前实现采用数组化的 `vruntime` 模型：

- 私有状态：`vruntime[]`、`weight[]`、`min_vruntime`。
- 默认权重：`1024`。
- 新进程初始化：继承当前系统内最小 `vruntime`，避免永远从 0 起跑。
- tick 更新：每运行 1 tick，就按 `delta = runtime * DEFAULT_WEIGHT / weight[idx]` 增加其 `vruntime`。
- 选择逻辑：优先挑选 `vruntime` 最小的 `RUNNABLE` 进程。
- tie-break：相同 `vruntime` 下选择 `ctime` 更早者。
- 退出更新：回收该进程槽位并重新计算系统的 `min_vruntime`。

这与 Linux 红黑树版 CFS 不同，但“最小虚拟运行时间优先”的核心思想已经具备，足以支撑实验里的公平性对比。

## 八、统计与用户态接口设计

### 8.1 用户态接口

当前调度相关系统调用包括：

- `setscheduler(int policy)`：切换当前全局调度策略。
- `getscheduler(void)`：获取当前策略。
- `setpriority(int pid, int priority)`：修改某个进程的优先级。
- `getstats()`：读取 CPU 统计信息。

### 8.2 统计模块职责

`sched_stats.c` 负责维护以下系统级统计：

- `total_ticks`
- `idle_ticks`
- `context_switches`
- `interrupts`
- `proc_created`
- `proc_exited`
- `syscalls`

同时还负责更新每个进程的：

- `rutime`
- `retime`
- `stime`

因此，调度实验、监控页面和测试框架都可以从这里获得统一的数据源。

## 九、关键设计取舍

### 9.1 为什么保留 `proc[]` 扫描而不实现复杂就绪队列

原因有三点：

1. xv6 原始结构就是数组扫描，改动成本最低。
2. `NPROC` 很小，O(n) 选择成本在实验平台上完全可接受。
3. 毕设重点是“多算法对比”和“框架化扩展”，不是追求生产级调度器的极致复杂度。

### 9.2 为什么用钩子而不是让每个策略直接改 `trap.c` / `proc.c`

因为钩子方案让系统保持以下优点：

- 生命周期接线集中、统一、可审计。
- 新增算法时只需要实现接口，不必改很多内核关键路径。
- 文档、测试和代码结构更一致。

### 9.3 为什么把高级调度器描述为“简化实现”

这是一个有意识的工程选择：

- xv6 没有复杂的进程实体、就绪树、负载均衡和多核公平机制。
- QEMU 单核教学环境更适合展示算法思想，而不是复刻工业级实现。
- 论文中如实写成“教学化、可运行、可验证的简化实现”，比夸大为“完整 Linux 级调度器”更稳妥。

## 十一、论文建议表述

如果用于毕设正文，建议采用以下口径：

- 本项目实现了一个基于 `sched_ops` 的可切换调度框架。
- 该框架通过 `proc_init / proc_exit / proc_tick / proc_wakeup` 四类生命周期钩子，把策略私有状态更新统一接入进程管理与时钟中断路径。
- FCFS、PRIORITY、SML、LOTTERY 用于展示经典静态或概率式调度思想；SJF、SRTF、MLFQ、CFS 则以教学化简化实现的方式展示更高级调度器的核心机制。
- 系统整体保留 xv6 原生 scheduler 外壳，只替换“策略选择器”和“策略状态更新器”，从而兼顾了实现复杂度、可解释性和实验可控性。

## 1. 模块定位

模块一是整个项目的基础层，负责把原始 xv6 从“单一 RR 调度 + 简单页分配”的教学基线，扩展为一个能够支撑算法比较、机制分析、回归测试、实验采样和监控展示的增强型内核。
如果你想先按更细的主题分开看，建议先读：

1. [模块一总览与边界](/home/niya/xv6/docs/module1/总览与边界.md)
2. [调度框架设计](/home/niya/xv6/docs/module1/调度框架.md)
3. [内存语义设计](/home/niya/xv6/docs/module1/内存语义.md)


如果只把模块一理解为“加了几种调度算法、做了几个内存特性”，会低估它真正的工程价值。更准确地说，模块一完成了两条主线上的系统化升级：

1. 调度主线：把原本只有一种策略的调度器，升级为可切换、可扩展、可观测的统一调度框架。
2. 内存主线：把原本相对线性的地址空间管理，升级为能同时承载 COW、Lazy Allocation 和匿名私有 `mmap` 的统一地址空间语义。

因此，模块一不是一组点状功能，而是整个系统的控制平面与内存语义基础。后续模块二测试、模块三实验、模块四监控，实际上都建立在模块一提供的统一接口和可观测能力之上。

## 2. 为什么要做这个模块

原始 xv6 的优势在于结构简洁、实现可教学，但这也天然带来两个明显限制。

第一，调度层过于单一。默认 Round-Robin 可以作为很好的一致性基线，但它无法表达批处理吞吐、交互响应、公平分配、优先级控制等不同调度目标，也无法支撑多策略对比实验。

第二，内存层过于朴素。原始地址空间模型适合说明页表、用户态和内核态的基本机制，但如果要进一步实现 COW、Lazy Allocation 和 `mmap`，若没有统一的地址空间元数据与缺页处理路径，多个特性就会变成互相割裂的补丁集合。

因此，模块一要解决的不是“如何多加几个功能”，而是两个更本质的问题：

1. 如何把多种调度策略纳入同一调度框架，使其可切换、可扩展、可观测。
2. 如何把多种地址空间机制纳入同一页表与缺页处理路径，使其能共存、可验证、可解释。

从这个角度看，模块一的目标是把 xv6 从教学原型推进到一个可比较、可验证、可实验的内核研究平台。

## 3. 模块目标

模块一的设计目标可以概括为以下四点：

1. 在 xv6/QEMU 的教学约束下实现多种可切换的调度策略。
2. 在不推翻 xv6 基本结构的前提下，引入更现代的内存管理能力。
3. 为模块二测试、模块三实验和模块四监控提供统一接口与观测点。
4. 将内核增强从“功能堆叠”提升为“可以写进论文的体系化设计”。

## 4. 当前实现范围

### 4.1 调度策略范围

当前内核支持 9 种策略：

| 策略 | 枚举 | 当前定位 |
|------|------|----------|
| DEFAULT / RR | `0` | xv6 基线轮转调度 |
| FCFS | `1` | 批处理对照策略 |
| PRIORITY | `2` | 静态优先级调度 |
| SML | `3` | 静态多级队列 |
| LOTTERY | `4` | 概率调度原型 |
| SJF | `5` | 基于历史运行时间估计的简化 SJF |
| SRTF | `6` | 基于剩余时间估计的抢占式原型 |
| MLFQ | `7` | 简化版多级反馈队列 |
| CFS | `8` | 基于 `vruntime` 的简化公平调度 |

### 4.2 论文主线中的内存管理范围

当前论文主线覆盖以下三条能力线：

1. Copy-on-Write `fork()`
2. Lazy Allocation
3. 匿名私有 `mmap/munmap` 原型

这三项能力并不是孤立并列的 feature，而是共同建立在统一地址空间元数据和统一缺页处理路径之上。

## 5. 调度算法是什么，分别解决什么问题

为了让后续架构设计形成完整闭环，首先需要说明这些调度算法本身代表什么思想，以及它们各自试图解决什么问题。

### 5.1 DEFAULT / Round-Robin

Round-Robin 是 xv6 的原生调度器。它为每个可运行进程轮流分配 CPU 时间片，强调基本公平性和实现稳定性。

它的价值主要体现在三点：

- 结构简单，是所有增强策略的基线对照组。
- 每个进程都有机会运行，不容易长期饥饿。
- 适合作为统一调度框架初始化时的默认策略。

### 5.2 FCFS

FCFS 按进入系统的先后顺序调度进程，在本项目中通过 `ctime` 判断先后。

它适合表达最简单的批处理思想：谁先来谁先执行。其优点是规则直接、解释成本低，但缺点是短任务可能被长任务阻塞，平均等待时间可能较差。

### 5.3 PRIORITY

优先级调度按照进程优先级选择下一个运行者。本项目中数值越小优先级越高，范围为 1 到 20。

这一类算法解决的是“任务重要性需要被显式表达”的问题。它适合做系统级控制，也适合作为测试和实验中的可控变量，但如果没有额外抑制机制，低优先级任务可能面临饥饿风险。

### 5.4 SML

SML 即静态多级队列。它把进程分配到多个固定优先级队列中，高层队列先被检查，队列内部再执行轮转。

它的重点不是单个优先级值，而是“任务类别”的静态分层。与 PRIORITY 相比，它更像固定层级的调度组织结构；与 MLFQ 相比，它不根据行为自动迁移。

### 5.5 LOTTERY

Lottery Scheduling 通过票数决定被调度的概率。票数越多，被选中的概率越大，但不是绝对压制其他进程。

它解决的是“如何在公平和优先之间引入比例控制”的问题。它很适合说明资源份额思想，也适合作为多策略实验中的概率型调度原型。

### 5.6 SJF

SJF 会优先执行预计运行时间最短的任务。它在理论上能够降低平均周转时间，是经典批处理优化算法。

但 SJF 最大的现实问题在于，系统事先并不知道任务会跑多久。因此，实现 SJF 的关键并不只是“选最短”，而是“如何估计谁更短”。

### 5.7 SRTF

SRTF 是 SJF 的抢占式版本。它持续比较当前各进程的剩余运行时间估计，一旦出现更短任务，就允许更积极地调整 CPU 分配。

它解决的是“短任务晚到达时如何不被长任务长期阻塞”的问题，因此更偏向于改善响应时间。

### 5.8 MLFQ

MLFQ 是操作系统中非常经典的通用调度思想。它根据进程行为动态调整优先级，让更像交互型的任务停留在高优先级队列，让更像 CPU 密集型的任务逐步下沉。

它的核心价值在于：系统不必事先知道任务类型，而是通过运行行为自动学习和反馈。

### 5.9 CFS

CFS 代表现代公平调度器思想。本项目采用 `vruntime` 近似表达“谁获得的 CPU 更少，谁应该优先运行”。

它解决的是“如何让长期 CPU 分配更加平滑、公平、连续”的问题。与简单 RR 相比，CFS 关注的不是瞬时轮转公平，而是累计公平。

## 6. 调度架构设计：从策略集合到统一控制平面

调度子系统最重要的设计成果，不是单个算法本身，而是统一调度框架的建立。

核心抽象定义在 [kernel/sched/sched.h](/home/niya/xv6/kernel/sched/sched.h) 中的 `struct sched_ops`：

```c
struct sched_ops {
    const char *name;
    void (*proc_init)(struct proc *p);
    struct proc* (*pick_next)(void);
    void (*proc_exit)(struct proc *p);
    void (*proc_tick)(struct proc *p);
    void (*proc_wakeup)(struct proc *p);
};
```

这个抽象把调度器定义为一组生命周期契约，而不是单个“挑下一个进程”的函数：

1. `proc_init` 负责建立策略相关状态。
2. `pick_next` 负责做核心调度决策。
3. `proc_exit` 负责进程退出时的状态清理或收尾。
4. `proc_tick` 负责 tick 级更新，如剩余时间、时间片、`vruntime` 等。
5. `proc_wakeup` 负责处理唤醒事件带来的状态调整。

这一设计有两个直接收益。

第一，调度框架与调度策略被清晰分离。框架只负责统一接线和调用，策略只负责自身规则。这样新增算法时不需要重写内核主流程。

第二，调度策略有了完整生命周期，而不是局限在 `scheduler()` 里的局部 if-else 分支。这使调度模块可以作为架构层成果来描述，而不仅是算法堆砌。

### 6.1 运行时切换机制

调度核心实现在 [kernel/sched/sched.c](/home/niya/xv6/kernel/sched/sched.c)。它维护：

- `current_policy`
- `current_ops`
- `sched_ops_table[]`

通过这些状态，系统可以在运行时完成策略切换，而不需要重启内核。切换流程为：

1. 校验新策略编号是否合法。
2. 从策略表中取得新的 `sched_ops`。
3. 调用 `sched_init_policy_state()` 为现有进程重建新策略所需状态。
4. 原子更新 `current_policy` 和 `current_ops`。

这使调度系统具备热切换能力，也为实验和课堂演示提供了很好的支撑。

### 6.2 生命周期钩子接线

统一框架还通过以下钩子把调度器接到内核关键路径上：

- `sched_proc_init_hook()`
- `sched_proc_exit_hook()`
- `sched_proc_tick_hook()`
- `sched_proc_wakeup_hook()`

因此，调度逻辑不再散落在多个路径里，而是通过统一接口接入 `allocproc()`、退出路径、时钟中断路径和唤醒路径。这样，SRTF、MLFQ、CFS 这类依赖运行时状态变化的算法，才能在一致框架下成立。

## 7. 调度实现：本项目是如何落地这些算法的

### 7.1 DEFAULT

`DEFAULT` 实现在 [kernel/sched/sched_default.c](/home/niya/xv6/kernel/sched/sched_default.c)。

实现中保留 `rr_index` 作为轮转扫描起点，使每次调度从上次结束的位置继续，这样可以避免进程表前部进程长期占优。

### 7.2 FCFS

FCFS 实现在 [kernel/sched/sched_fcfs.c](/home/niya/xv6/kernel/sched/sched_fcfs.c)。

其核心逻辑是遍历所有 `RUNNABLE` 进程，选出 `ctime` 最小者。这个实现简单，但能很好地展示“按到达顺序调度”与 RR 的差异。

### 7.3 PRIORITY

PRIORITY 实现在 [kernel/sched/sched_priority.c](/home/niya/xv6/kernel/sched/sched_priority.c)。

算法在所有 `RUNNABLE` 进程中选择优先级最高者，若优先级相同，则继续按 `ctime` 打破平局。配合 `setpriority()`，该策略具有良好的可控性和可验证性。

### 7.4 SML

SML 实现在 [kernel/sched/sched_sml.c](/home/niya/xv6/kernel/sched/sched_sml.c)。

实现中为不同 class 维护各自的 `rr_index`，调度时总是先检查高层队列，再检查低层队列，队列内部执行 RR。

### 7.5 LOTTERY

LOTTERY 实现在 [kernel/sched/sched_lottery.c](/home/niya/xv6/kernel/sched/sched_lottery.c)。

其基本过程是：

1. 统计所有可运行进程的总票数。
2. 生成一个随机中奖彩票号。
3. 让票区间覆盖该号码的进程获得 CPU。

该实现使用简单线性同余随机数生成器，重点不在高质量随机性，而在“份额公平”的可解释原理。

### 7.6 SJF

SJF 实现在 [kernel/sched/sched_sjf.c](/home/niya/xv6/kernel/sched/sched_sjf.c)。

由于系统无法预知作业长度，本项目采用教学化估计模型，维护：

- `estimated_time[]`
- `actual_time[]`
- `burst_count[]`

在进程退出时，用历史平均值修正运行时间估计，并以旧值与新值折中的方式更新估计结果。这样，SJF 才从一个理论概念转化为可执行的近似策略。

### 7.7 SRTF

SRTF 实现在 [kernel/sched/sched_srtf.c](/home/niya/xv6/kernel/sched/sched_srtf.c)。

相比 SJF，它额外维护 `remaining_time[]`。每个 tick 通过 `proc_tick` 钩子递减剩余时间，调度时总是选择剩余时间估计最短的进程，平局时再用 `ctime` 打破。

这让系统在短任务后到达时，也能更积极地把 CPU 让给更短任务，从而改善响应时间。

### 7.8 MLFQ

MLFQ 实现在 [kernel/sched/sched_mlfq.c](/home/niya/xv6/kernel/sched/sched_mlfq.c)。

当前采用三层队列：

- 队列 0：时间片 10
- 队列 1：时间片 20
- 队列 2：时间片 40

核心状态包括：

- `queue_level[]`
- `time_in_queue[]`
- `rr_index[]`

反馈机制为：

- 新进程进入最高优先级队列。
- 持续占用 CPU 达到该层时间片后被降级。
- 睡眠后再次唤醒则提升一级并清空该层已消耗时间。

这套逻辑体现了通过行为反馈动态识别任务类型的思想。

### 7.9 CFS

CFS 实现在 [kernel/sched/sched_cfs.c](/home/niya/xv6/kernel/sched/sched_cfs.c)。

实现中为每个进程维护 `vruntime[]`，每运行一个 tick 就更新一次虚拟运行时间。下一次调度时，总是选择 `vruntime` 最小的可运行进程。

这意味着“累计获得 CPU 更少”的进程会优先运行，从而把公平性显式量化为一个可比较的时间指标。

## 8. 内存管理架构设计：从 feature 拼接到统一地址空间语义

模块一的内存设计不是简单为 xv6 添加多个独立特性，而是围绕统一地址空间语义建立起来的。

它主要包含两层论文主线状态：

1. 进程地址空间层：由 [kernel/proc.h](/home/niya/xv6/kernel/proc.h) 中的 `heap_end`、`vmas[]` 和 `sz` 共同描述。
2. 页表与缺页处理层：由 [kernel/vm.c](/home/niya/xv6/kernel/vm.c) 与 [kernel/sysproc.c](/home/niya/xv6/kernel/sysproc.c) 完成真实映射、缺页分配与权限控制。

其中：

- `heap_end` 表示堆的真实边界。
- `vmas[]` 描述匿名私有映射区域。
- `sz` 表示当前地址空间最高边界。

这种划分的关键意义在于，进程的“逻辑有效地址空间”与“当前已经建好页表映射的物理存在”不再完全绑定于单一 `sz`，从而为懒分配、COW 和 `mmap` 并存打下基础。

## 9. 内存实现主线：COW、Lazy、mmap 如何协同

### 9.1 Copy-on-Write

COW 让 `fork()` 后的父子进程首先共享物理页，仅在写入发生时再创建私有副本。这显著降低了大量 `fork` 后立即 `exec` 或只读访问场景下的内存复制成本。

更重要的是，它要求系统建立统一的页权限与写时复制语义，不能只在用户态缺页时处理，而忽略内核代用户写回等路径。

### 9.2 Lazy Allocation

Lazy Allocation 的思想是：地址空间可以先被声明，物理页等到首次访问时再真正分配。

这使堆扩展成本从“申请时立即付出”转为“访问时按需付出”，也是 xv6 从教学型 eager 分配走向更现代地址空间语义的重要一步。

### 9.3 匿名私有 mmap

本项目的 `mmap/munmap` 是教学化原型，目前只支持匿名私有映射，即 `MAP_PRIVATE | MAP_ANONYMOUS` 组合。

它的重点不在完整 POSIX 兼容，而在于把“按区间描述地址空间”的思想正式引入 xv6，为缺页映射和地址空间拆分回收建立更清晰的语义基础。

### 9.4 统一缺页路径

在统一设计下：

1. 用户通过 `sbrk` 或 `mmap` 扩展地址空间，系统先更新元数据。
2. 当用户首次访问尚未建页的合法地址时，`usertrap()` 转入缺页处理。
3. 缺页处理统一判断 fault 地址是属于堆、VMA 还是非法区域。
4. 对于 COW 页，通过专门复制路径建立私有副本。
5. 对于 `munmap()` 或地址空间收缩，再同步回收页表和元数据。

这就把 Lazy Allocation、COW 和 `mmap` 从并列补丁，统一收束为同一地址空间模型下的不同语义分支。

模块一补充了多类关键接口：

这些接口的意义不只是“功能可调用”，更重要的是把模块一变成了可验证、可采样、可展示的系统基础。

## 11. 关键设计取舍

### 11.1 为什么采用统一调度框架，而不是在 `scheduler()` 中堆分支

因为很多调度策略并不只是“选谁先运行”这么简单。SRTF、MLFQ 和 CFS 都依赖创建时间、剩余时间、唤醒行为或虚拟运行时间等额外状态。如果这些逻辑散落在 `scheduler()` 或系统调用分支中，后续测试、调试和论文描述都会变得混乱。

统一框架把“策略是生命周期契约”这一点显式化，使调度模块真正具备架构层表达能力。

### 11.2 为什么采用统一缺页路径，而不是为每个 feature 单独写 fault 逻辑

如果 Lazy Allocation、COW 和 `mmap` 各自维护自己的 fault 路径，最终一定会出现地址合法性判断冲突、权限语义不一致或回收策略脱节的问题。

统一缺页入口的价值在于，所有地址解释、权限落地和页表更新都遵循同一套规则。

### 11.3 为什么保留教学化简化，而不追求工业级完整实现

本科毕设的重点是构建一个可解释、可验证、可实验的增强系统，而不是无限堆高复杂度。

因此，本项目保留了若干有意识的边界，例如：

- `SJF/SRTF` 是基于历史的教学化估计模型，而不是工业级 burst 预测系统。
- `MLFQ` 是简化版多级反馈队列，没有完整全局 boost 机制。
- `mmap` 只支持匿名私有映射，不支持文件映射、共享映射和回写。

这些边界并不意味着设计失败，反而体现了在教学系统中对复杂度与表达价值的平衡。

## 13. Bug 如何推动方案收敛

模块一不是一步到位的，它的最终架构实际上是通过若干关键 bug 暴露问题后逐步收敛出来的。

### 13.1 调度器切换一致性问题

调度器切换相关问题说明，策略切换不是普通配置改动，而是共享控制平面的一部分。策略切换必须先初始化新策略状态，再受控切换当前策略，否则就会出现一致性与竞态风险。

### 13.2 `copyout` 与 COW 语义不一致

该问题说明，写时复制语义不能只覆盖用户态写缺页，内核代用户写回同样必须遵守一致的页表语义。这个问题直接推动了模块一从“缺页补丁集合”走向“统一页表语义设计”。

### 13.3 接口变更引发的启动异常

这类问题说明，模块一的风险并不只来自局部算法，而来自接口契约变化后全系统的连锁影响。因此，模块一后续文档和验证策略更强调统一接口、最小回归集和生命周期接线完整性，而不是只强调单点实现。

## 14. 验证闭环：如何证明模块一真正成立

模块一的价值不能只靠“代码存在”来证明，而必须通过一套完整闭环来支撑。

### 14.1 调度侧闭环

调度侧目前已经形成以下闭环：

提出调度目标 -> 设计/选择策略 -> 通过 `sched_ops` 接入统一框架 -> 通过 `setscheduler()` 在运行时切换 -> 通过 `getptable()`、统计字段和用户态程序观察行为 -> 通过模块二测试与模块三实验验证效果。

### 14.2 内存侧闭环

内存侧目前已经形成以下闭环：

提出地址空间机制需求 -> 通过 `heap_end + vmas[] + sz` 描述合法空间 -> 通过统一缺页路径落实 Lazy/COW/`mmap` 语义 -> 通过 `getmemstat()`、专项测试和实验脚本采样观察 -> 通过 `munmap`、回收路径和边界测试验证一致性。

### 14.3 三层证据体系

模块一的验证由三层证据共同支撑：

1. 模块二的回归测试，如 `_test_runner` 中的内存、边界与调度断言。
2. 专项用户态程序，如 `scheduler_test`、`mmap_test`、`test_sjf`、`test_srtf`、`test_mlfq`、`test_cfs`。
3. 模块三正式实验中的结构化采样、聚合与统计观察。

这意味着模块一已经不是“若干 feature 可以跑起来”，而是已经形成了架构、接口、验证和实验相互支撑的完整闭环。

## 15. 最终结论

模块一已经不再是“实现了很多点状 feature”，而是形成了一个可解释、可测试、可实验的内核增强模块。

从架构角度看，它完成了三层升级：

1. 在调度层，建立了支持多策略切换的统一调度框架。
2. 在内存层，建立了支持 COW、Lazy 和匿名私有 `mmap` 共存的统一地址空间语义。
3. 在系统层，为测试、实验和监控提供了统一接口和观测入口。

因此，更准确的论文表述应当是：模块一通过统一调度框架和统一地址空间语义，把原始 xv6 扩展为一个可比较、可验证、可实验的内核研究平台，而不是简单为其添加若干彼此孤立的功能。

# 模块一：内存管理优化架构设计（对齐当前实现）

## 一、文档说明

虽然文件名沿用 `2.内存管理优化.md`，但从毕设模块划分上看，这部分内容属于模块一，与调度算法优化共同构成“内核机制增强”部分。本文档描述的是当前仓库里已经落地的内存管理实现，而不是最初设计稿。

对齐基线：`6a343ce` (`fix(module1): close scheduler and vm integration gaps`)

本文档覆盖的代码主要位于：

- `kernel/kalloc.c`
- `kernel/vm.c`
- `kernel/vm.h`
- `kernel/proc.h`
- `kernel/proc.c`
- `kernel/exec.c`
- `kernel/sysproc.c`
- `kernel/trap.c`

## 二、设计目标与约束

本模块关注四类能力：

1. Buddy System：减少外部碎片，提升大块可用性。
2. Copy-on-Write：降低 `fork()` 的立即复制开销。
3. Lazy Allocation：把“保留地址空间”和“实际分配物理页”分离。
4. `mmap/munmap`：提供匿名私有懒映射原型。

但这些能力不是在 Linux 环境中实现，而是在 xv6/QEMU 的教学约束下实现，因此从一开始就带有几个设计边界：

- 仅考虑 Sv39 页表和 4KB 页。
- 仅支持匿名私有映射，不支持文件映射、共享映射、固定地址映射。
- 进程地址空间仍尽量复用 xv6 原始 `sz` 语义，避免全盘重写 `exec/fork/free` 路径。
- 优先保证架构闭环和实验可观测性，再考虑完整 POSIX 兼容性。

## 三、总体架构

### 3.1 分层结构

```text
模块一内存管理子系统
├── 物理页分配层
│   └── kernel/kalloc.c
│       ├── Buddy System
│       ├── 物理页引用计数
│       └── 内存统计计数器
├── 进程地址空间元数据层
│   ├── kernel/proc.h
│   ├── kernel/proc.c
│   └── kernel/exec.c
│       ├── heap_end
│       ├── vmas[NVMA]
│       ├── proc_mmap_limit()
│       └── proc_vm_top()
├── 虚拟内存执行层
│   └── kernel/vm.c
│       ├── uvmcopy() / cow_alloc()
│       ├── vmfault()
│       ├── copyin/copyout/copyinstr
│       └── uvmalloc/uvmdealloc/uvmunmap
└── 用户接口层
    └── kernel/sysproc.c
        ├── sbrk(eager/lazy)
        ├── mmap()
        ├── munmap()
        └── getmemstat()
```

### 3.2 设计主线

当前实现不是四个完全独立的功能，而是一条统一路径：

- Buddy System 负责“页从哪里来”。
- COW 负责“什么时候需要真正复制页”。
- Lazy Allocation 负责“什么时候需要真正分配页”。
- `mmap` 负责“哪些虚拟地址区间允许按需建页以及对应权限是什么”。
- `vmfault()` 则是把上面三者串起来的核心总入口。

因此，本模块最重要的架构不是某个单一算法，而是“统一缺页路径”。

## 四、进程地址空间模型

### 4.1 从单一 `sz` 到 `heap_end + vmas + sz`

原始 xv6 通常只用 `p->sz` 代表“用户地址空间上界”，这在只有代码段、数据段、栈和线性堆时够用。但一旦同时引入：

- 懒分配的 heap
- `mmap` 映射区
- `munmap` 产生的空洞

单一 `sz` 就不够表达“哪些地址合法但尚未建页”“哪些地址曾经映射过但已经失效”“哪些地址属于堆，哪些地址属于映射区”。

因此当前实现将地址空间元数据拆成了三层：

```c
uint64 sz;          // 当前地址空间最高边界（heap 或 vma 的最大终点）
uint64 heap_end;    // 线性堆的逻辑终点
struct vma vmas[NVMA];
```

### 4.2 `struct vma`

```c
struct vma {
  uint64 addr;
  uint64 length;
  int prot;
  int flags;
  int valid;
};
```

当前 `vma` 只服务匿名私有映射，但已经具备一个最小可用的映射描述能力：

- `addr`：映射起始虚拟地址。
- `length`：映射长度，按页对齐。
- `prot`：记录读写权限，供缺页时转成 PTE 权限。
- `flags`：记录 `MAP_PRIVATE | MAP_ANONYMOUS` 等标志。
- `valid`：标记槽位是否被占用。

### 4.3 相关辅助函数

为了让地址空间操作保持清晰，`proc.c` 中增加了几组辅助函数：

- `proc_vm_init(p)`：初始化 `heap_end` 和 `vmas[]`。
- `proc_find_vma(p, va)`：查询某个地址是否落在有效 VMA 中。
- `proc_mmap_limit(p)`：返回堆允许扩展到的上限，防止堆和映射区撞车。
- `proc_vm_top(p)`：重新计算整个进程地址空间的最高边界。

这几个函数是本模块架构设计里的关键，因为它们把“地址空间判断规则”集中起来了，而不是散在 `sbrk/mmap/munmap/vmfault` 各处重复写。

### 4.4 `fork` 和 `exec` 如何维护地址空间元数据

- `kfork()`：除了复制页表和寄存器，还会复制 `heap_end` 和 `vmas[]`。
- `kexec()`：加载新程序后，把 `heap_end` 设为新镜像大小，并把 `vmas[]` 清空。
- `freeproc()`：释放进程时调用 `proc_vm_init()`，避免槽位复用后残留旧 VMA。

这保证了地址空间元数据与进程生命周期保持一致。

## 五、物理内存管理：Buddy System

### 5.1 核心数据结构

当前物理页分配器位于 `kernel/kalloc.c`：

```c
#define MAX_ORDER 10

struct buddy_free_list {
  struct run *head;
};

struct {
  struct spinlock lock;
  struct buddy_free_list free_lists[MAX_ORDER];
} kmem;
```

含义如下：

- `order = 0` 表示 1 页块。
- `order = 1` 表示 2 页块。
- `order = 9` 表示 512 页块。
- 最大块大小约为 `2^10 * 4KB = 4MB`。

### 5.2 分配过程

Buddy 的分配入口是 `buddy_alloc(order)`，`kalloc()` 实际调用 `buddy_alloc(0)` 获取 1 页。分配流程为：

1. 从目标 `order` 开始，向上寻找第一个非空闲链表。
2. 若找到更大的块，则不断向下分裂。
3. 每次分裂把高地址半块挂回下一层空闲链表。
4. 最终返回一个满足大小要求的块。
5. 对返回块设置引用计数 `ref_count[idx] = 1`。

### 5.3 释放过程

释放入口是 `kfree()`，内部调用 `buddy_free(pa, 0)`：

1. 先把物理页内容用 `1` 填充，帮助暴露悬空引用。
2. 清空该页的引用计数。
3. 尝试计算同阶 buddy 地址。
4. 若 buddy 当前也空闲，则从空闲链表中摘掉它并与当前块合并。
5. 不断重复直到无法继续合并或达到最大阶。
6. 把最终块挂回合适的空闲链表。

### 5.4 统计信息

`kalloc.c` 维护了以下统计量：

- `total_pages`
- `free_pages`
- `buddy_merges`
- `buddy_splits`
- `cow_faults`
- `lazy_allocs`
- `cow_copy_pages`

这些统计数据通过 `getmemstat()` 提供给用户态测试和监控模块。

### 5.5 当前实现的取舍

Buddy 当前实现仍采用 `is_on_free_list()` 在线性扫描链表中判断某个 buddy 是否空闲，因此在理论上不是最优复杂度实现。但对于 xv6 的物理内存规模和本科毕设目标，这个取舍是合理的：

- 逻辑直观，便于讲解和答辩。
- 足以支撑数据采集和实验对比。
- 没有引入过于复杂的位图或伙伴树结构。

## 六、Copy-on-Write 设计

### 6.1 设计目标

COW 的目标是优化 `fork()`：

- `fork()` 时不立即复制物理页。
- 父子进程共享同一组物理页。
- 真正发生写入时才分配新页并复制内容。

### 6.2 `uvmcopy()` 中的 COW 建立过程

当前 `uvmcopy()` 的做法是：

1. 遍历父进程地址空间中已经映射的页。
2. 若页本来可写，则：
   - 清除 `PTE_W`
   - 设置 `PTE_COW`
   - 父页表项改为只读共享
3. 子页表映射到同一物理页，权限与父页相同。
4. 对共享物理页执行 `ref_inc(pa)`。

这意味着：

- 未实际分配的 lazy 页面不会被强行建页。
- `fork()` 后父子页表共享同一份物理内存。

### 6.3 写时复制：`cow_alloc()`

发生写 fault 时：

1. 通过 `walk()` 找到对应 PTE。
2. 校验该页确实是有效的 `PTE_COW` 页面。
3. 分配新物理页。
4. 把旧页内容复制到新页。
5. 清除 `PTE_COW`，恢复 `PTE_W`。
6. 刷新该页的 TLB。
7. 对旧页执行 `ref_dec(pa)`，如果引用归零则释放。
8. 更新 `cow_faults` 和 `cow_copy_pages`。

### 6.4 不只在用户态 fault 路径处理 COW

一个很重要的设计细节是：COW 不只在 `usertrap()` 中处理。

`copyout()` 也会在发现目标页为只读 COW 页时调用 `cow_alloc()`。这是必要的，因为 xv6 内核常常需要把数据写回用户缓冲区，例如：

- `wait()` 把退出状态写回用户地址。
- 各类系统调用把结果写回用户缓冲区。

如果 `copyout()` 不认识 COW，那么“用户地址合法，但页面是共享只读页”的场景会被误判成写失败。

## 七、Lazy Allocation 设计

### 7.1 `sbrk` 的两种模式

当前 `sys_sbrk()` 支持两种模式：

- `SBRK_EAGER`：立即分配物理页。
- `SBRK_LAZY`：仅扩展 `heap_end`，不立即分配页。

当采用懒分配时：

1. 返回旧的 `heap_end` 作为起始地址。
2. 只增长 `heap_end`。
3. 必要时更新 `p->sz`。
4. 真正访问页面时再由 `vmfault()` 完成建页。

### 7.2 为什么要保留 `growproc()`

`growproc()` 仍然保留，是因为：

- 负增长 `sbrk` 需要真正回收物理页。
- eager 模式需要沿用 xv6 的 `uvmalloc/uvmdealloc` 逻辑。
- 这样能最大限度复用原始内核代码。

### 7.3 `vmfault_region()`：统一判断 fault 属于哪一类区域

`vm.c` 里的 `vmfault_region()` 是当前实现的关键：

1. 若 `va < PGSIZE`，拒绝访问空指针区域。
2. 若 `va < heap_end`，说明这是 heap 的 lazy 页面，直接赋予 `PTE_U | PTE_R | PTE_W`。
3. 否则尝试用 `proc_find_vma()` 查询 VMA。
4. 若命中 VMA，则根据 `prot` 判断当前访问是否合法，并生成对应页表权限。

因此，heap 懒分配和 `mmap` 懒映射已经被收敛到同一个权限决策函数里。

## 八、`mmap/munmap` 设计

### 8.1 当前支持范围

当前 `mmap` 支持的能力是：

- `MAP_PRIVATE`
- `MAP_ANONYMOUS`
- `PROT_READ`
- `PROT_WRITE`
- 按页对齐
- 懒分配建页
- `munmap` 释放完整映射、头部、尾部或中间子区间

不支持的能力包括：

- 文件映射
- `MAP_SHARED`
- 固定地址映射
- 页级脏页回写
- 完整 POSIX 语义

### 8.2 `sys_mmap()` 的工作流程

1. 校验 `flags` 必须同时包含 `MAP_PRIVATE | MAP_ANONYMOUS`。
2. 校验 `fd == -1` 且 `offset == 0`。
3. 校验 `prot` 至少包含 `PROT_READ` 或 `PROT_WRITE`。
4. 长度向上按页对齐。
5. 在 `vmas[]` 中分配一个空槽位。
6. 以 `PGROUNDUP(p->sz)` 作为新映射起点。
7. 将 `addr/length/prot/flags/valid` 写入该 VMA。
8. 更新 `p->sz` 为映射后的新上界。

注意：这一步不会立即调用 `kalloc()`，只是登记一个合法映射区间。

### 8.3 `sys_munmap()` 的工作流程

当前 `munmap()` 支持 4 种情形：

1. 整段 VMA 全部释放。
2. 释放 VMA 头部。
3. 释放 VMA 尾部。
4. 释放 VMA 中间一段，此时需要把原 VMA 拆成两段。

处理步骤为：

1. 遍历 `vmas[]` 找到完全覆盖 `[addr, addr+length)` 的 VMA。
2. 若释放的是中间子区间，则先预留一个新的 VMA 槽位。
3. 调用 `uvmunmap()` 释放已经实际映射的物理页。
4. 根据释放区间与原 VMA 的相对位置，修改原 VMA 或生成新的尾部 VMA。
5. 通过 `proc_vm_top()` 重新计算 `p->sz`。

这也是当前版本相比旧版设计的一个关键升级：

- 旧版只有“扩 `sz`”没有 VMA 记录。
- 现在 `munmap` 之后留下的空洞不会再被误当成合法 heap 页面重新懒分配出来。

### 8.4 `prot` 权限如何真正落到页表上

`vm.c` 中的 `vma_prot_to_perm()` 会把：

- `PROT_READ` 转成 `PTE_R | PTE_U`
- `PROT_WRITE` 转成 `PTE_W | PTE_R | PTE_U`

当 VMA 页面第一次 fault 时，`vmfault()` 会根据 VMA 的 `prot` 使用对应权限建页。也就是说：

- `mmap(..., PROT_READ, ...)` 不会被当作可写页。
- 写一个只读映射会在 fault 检查阶段被拒绝。

## 九、统一缺页处理路径

### 9.1 `usertrap()` 中的 fault 处理顺序

当前 `trap.c` 中，用户态页错误只处理：

- `scause == 13`：Load page fault
- `scause == 15`：Store/AMO page fault

处理顺序是：

1. 先检查当前页表项是否是有效的 `PTE_COW` 页面。
2. 若是，则调用 `cow_alloc()`。
3. 否则调用 `vmfault()`。
4. 若仍失败，则打印错误并 kill 当前进程。

### 9.2 `vmfault()` 的职责边界

`vmfault()` 只负责两件事：

1. 判定 fault 地址是否属于“允许按需建页”的区域。
2. 若合法，则分配新页并以正确权限映射。

它不直接区分“这是堆还是 `mmap`”的高层语义，而是把判断委托给 `vmfault_region()`。这一点让架构更清晰。

### 9.3 `copyin/copyout/copyinstr` 统一走 fault 路径

当前实现中：

- `copyout()` 在目标页不存在时会触发 `vmfault()`；若遇到 COW 页则触发 `cow_alloc()`。
- `copyin()` 在源页不存在时也会触发 `vmfault()`。
- `copyinstr()` 同样如此。

这保证了一个很关键的性质：

- 即使某块用户缓冲区是合法但尚未分配的 lazy 页面，系统调用仍然可以在第一次访问时把它建出来。
- 这让“懒分配 + 系统调用写回用户缓冲区”成为可工作的组合，而不是一进入内核就失败。

## 十一、关键设计取舍与剩余限制

### 11.1 为什么不把 `mmap` 做成完整 POSIX 版本

原因很现实：

- xv6 原本没有 VFS 层面的页缓存与文件回写机制。
- 完整文件映射和共享映射会显著抬高实现复杂度。
- 对本科毕设来说，匿名私有懒映射已经足以展示“地址空间元数据 + fault 权限判定 + 懒建页”这条完整链路。

### 11.2 为什么保留 `MAX_PAGES = 40000`

当前 `PHYSTOP = 128MB`，对应页数为 `32768`，因此 `40000` 仍在安全范围内。这个上界不是最优雅的写法，但在当前硬件模型下足够使用。

### 11.3 当前仍然应该诚实说明的限制

- `mmap` 仅支持匿名私有映射。
- `Buddy` 的 buddy 空闲检测仍是链表扫描实现。
- 地址空间模型已经比旧版完整，但仍是 xv6 规模下的最小可用 VMA 设计。
- 这套实现适合论文中的“教学化、可运行、可测量原型”，不宜表述为工业级内存子系统。

## 十二、论文建议表述

建议在正文中使用如下口径：

- 本项目将物理页分配、COW、懒分配和匿名私有 `mmap` 收敛到统一的缺页处理框架中。
- 通过把 `sz` 拆解为“地址空间上界 `sz` + 线性堆终点 `heap_end` + VMA 数组 `vmas[]`”，系统能够同时表达 heap 扩展、映射区间和 `munmap` 后空洞。
- `vmfault()` 不直接绑定某一种机制，而是通过 `vmfault_region()` 统一判断 fault 属于 heap 还是 VMA，并把 `prot` 权限真正映射为页表权限。
- Buddy、COW、Lazy、mmap 四者并不是平行堆砌，而是通过统一页分配器、统一页错误路径和统一统计出口形成一个完整的内存管理增强架构。

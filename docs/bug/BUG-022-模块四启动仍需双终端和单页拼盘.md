
# BUG-022: 模块四启动仍需双终端和单页拼盘

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-022 |
| 分类 | 用户工具 (USER) |
| 严重程度 | 高 |
| 初次记录时间 | 2026-03-26 |
| 影响文件 | `dashboard/scripts/start_monitor.sh`, `dashboard/scripts/run_dashboard.sh`, `dashboard/scripts/capture_xv6_console.sh`, `dashboard/index.html`, `dashboard/js/router.js`, `dashboard/js/views/home.js` |

## 问题描述

模块四在这次收口之前，入口和界面其实是两层问题叠在一起：

1. 启动链要靠两个终端手工拼接，用户必须先开 bridge，再另外去跑 xv6/QEMU，默认体验不是“一条命令拉起监控器”。
2. 前端还是单页拼盘，主页、进程、调度、内存、事件和说明都堆在一个大页面里，缺少真正的主页 -> 子页面 -> 返回主页结构。

从使用者角度看，这会让模块四更像“调试工具集合”，而不是毕设里想表达的教学任务管理器。

## 复现现象

修复前的典型行为是：

```text
终端 A: 启动 bridge
终端 B: 再手工启动 xv6/QEMU
```

页面侧则是：

- 主页和各子视图混在一起。
- 入口信息和实时监控信息挤在同一层。
- 没有明确的页面路由边界。

## 排查过程

1. 先看 `run_dashboard.sh`，发现它只是 bridge-only 启动脚本，并没有把 xv6/QEMU 生命周期一并托管起来。
2. 再看前端实现，发现 `index.html` 和 `app.js` 仍是单页拼装，页面职责没有拆开。
3. 对照模块四审查文档，确认问题本质不是“某个按钮没摆好”，而是整个入口和页面结构都还停留在工程样机阶段。
4. 结合后续提交 `078866e`，确认真正的修复方向是把启动入口和前端结构一起收口，而不是只改一个页面样式。

## 根因结论

根因是模块四早期把“桥接启动”“控制台分流”和“前端展示”混在一条链上，缺少统一入口，也缺少页面级路由拆分。

这会直接导致两个后果：

- 用户必须记住两条命令，体验割裂。
- 前端只能靠单页堆区块去模拟导航，无法形成教学任务管理器式的结构。

## 修复方案

1. 新增 `dashboard/scripts/start_monitor.sh` 作为默认入口，统一托管 bridge 和 xv6/QEMU 生命周期。
2. 将 `run_dashboard.sh` 降级为 bridge-only 辅助脚本，把 `capture_xv6_console.sh` 明确标为 fallback/debug。
3. 将前端重构为路由化结构，拆出主页、进程、调度、内存和事件等子页面。
4. 将首页收束成论文导览 + live monitor 入口，不再把所有信息堆成一个大面板。

## 修复后验证

修复后已确认：

- `start_monitor.sh` 可以作为默认启动入口。
- bridge 能后台托管，xv6/QEMU 可以直接前台交互。
- 前端已经具备页面级路由和子页面结构。
- `capture_xv6_console.sh` 只作为 fallback/debug 使用。

## 经验结论

1. 教学 OS 的监控界面不能只追求“能显示”，还要追求“入口清楚”。
2. 启动脚本、桥接链路和前端页面必须一起设计，不然体验永远会分裂。
3. 单页拼盘能凑出 demo，但很难支撑论文里的任务管理器叙事。

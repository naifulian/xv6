# BUG-019: 调度实验场景区分度不足

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-019 |
| 分类 | 测试模块 (TEST) |
| 严重程度 | 中 |
| 初次记录时间 | 2026-03-22 |
| 复核日期 | 2026-03-25 |
| 影响文件 | `user/perftest.c`, `webui/validate_logs.py`, `webui/collect_data.sh` |

## 现象与触发条件

在旧版 scheduler 实验里，`convoy/fairness/response` 多个场景经常一起塌缩到 `1-3 ticks`。如果只看表格，很容易误读成：

- 调度器差别不大；
- 实验结果很稳定；
- 某些算法“性能几乎一样”。

但真正的问题不是调度器完全一样，而是工作负载太轻、组织方式太粗，导致实验分辨率不够。

## 影响范围

1. 调度结论缺乏说服力。
2. validator 只能看到“数字存在”，但旧版本不一定能分辨这些数字其实没有区分度。
3. formal dataset 即使补轮次，也可能只是把低分辨率结果重复 3 次。

## 复现与观察

旧日志最直接的症状是：

- 多个不同策略在多个场景里拿到完全相同的 tick 数；
- fairness/response 后半段甚至会因为单个长命令过重而超时；
- scheduler 一轮里前半段能跑，后半段经常缺项。

## 排查过程

1. 先从 validator 的 low-resolution 和场景 warning 入手，确认问题不是个别轮次偶然抖动。
2. 回看 `perftest sched` 的组织方式，发现多个场景被捆在一个长命令里，后半段容易超时。
3. 进一步回看每个子场景的 work/batch/service budget，发现若 workload 太轻，即使算法不同也只会落在 1-3 tick 量级。
4. 最后将问题拆成两个层面：
   - 场景组织需要 split；
   - workload/batch 需要调到能稳定拉开差异的量级。

## 根因结论

根因不是“某个调度器坏了”，而是实验设计本身分辨率不足：

- workload 太轻；
- 命令粒度太粗；
- 某些场景的停止条件和测量方式会掩盖算法差异；
- 在 xv6/QEMU 单核 tick 环境下，这些问题会被进一步放大。

## 修复方案

1. 将 testing 的 scheduler 正式采集拆成 `sched_core / sched_fairness / sched_response`。
2. 调整 fairness、response、convoy 等场景的工作负载和 batch，让结果离开 `1-3 ticks` 危险区间。
3. 在 validator 里继续保留 low-resolution 和方法学 warning，避免未来再次静默退化。
4. 对旧数据保持审计用途，但不再把它当作正式结论来源。

## 修复后验证

修复后应能看到：

- fairness 和 response 不再在中途缺项；
- `SCHED_CONVOY_RR`、`SCHED_RESPONSE_MLFQ` 等指标离开低分辨率区间；
- testing 可以形成完整 formal round，而不是只有 memory 部分稳定。

## 为什么这不是误判

如果算法真的没有差异，那么在提升 workload 和分辨率后，结果仍应继续收敛在一起。但本轮重构后，多个场景已经能稳定拉开差别，说明旧问题确实在实验设计，不在“算法天然一样”。

## 经验结论

1. “结果一样”不总是系统结论，也可能是实验没测出来。
2. 对教学 OS，场景设计必须显式考虑 tick 分辨率和 QEMU 噪声。
3. 模块三真正难的不是画图，而是先保证图背后的实验有区分度。

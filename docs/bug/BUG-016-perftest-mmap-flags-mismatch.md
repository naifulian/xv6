# BUG-016: `perftest` 的 `mmap` 参数与内核实现不匹配

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-016 |
| 分类 | 测试模块 (TEST) |
| 严重程度 | 高 |
| 初次记录时间 | 2026-03-22 |
| 复核日期 | 2026-03-25 |
| 影响文件 | `user/perftest.c`, `kernel/sysproc.c`, `webui/validate_logs.py` |

## 现象与触发条件

模块三早期数据里，`MMAP_SPARSE` 和 `MMAP_FULL` 曾长期出现 `-1` 或 error 结果。最开始如果只看图表，很容易把它误读成“mmap 性能很差”或“某个场景极度退化”。

但这类结果的真正含义不是“性能差”，而是测试根本没有按照内核当前支持的接口跑起来。

## 影响范围

1. `MMAP_*` 相关实验结果失效。
2. 模块三会把失败场景混进后续分析链路。
3. 如果校验器不够严格，这类错误甚至可能继续流入论文图表和 Web 展示。

## 复现与观察

排查时观察到两件事：

1. `perftest` 里的 `mmap` 调用只传了 `MAP_ANONYMOUS` 风格的 flags。
2. 当前 xv6 的 `sys_mmap()` 实现明确要求 `MAP_PRIVATE | MAP_ANONYMOUS` 同时出现。

也就是说，旧测试代码和当前内核实现的约束根本不在一个协议上。

## 排查过程

1. 先从结果层发现 `MMAP_*` 经常变成 `-1`，说明这不是正常时间测量值。
2. 回到 `perftest.c`，发现 `mmap` 调用参数与当前 `sys_mmap()` 支持集不一致。
3. 再对照内核对 flags 的检查逻辑，确认问题不是“性能抖动”而是“调用条件不满足”。
4. 最后结合模块三校验器，确认旧链路没有把这类失败当成硬错误，这也是它能继续污染出图结果的原因之一。

## 根因结论

根因是实验侧接口假设没有跟内核实现同步：

- `perftest` 仍按旧口径或不完整口径调用 `mmap`；
- 当前内核实现要求更严格的 flags 组合；
- 结果就是实验根本没有进入有效测量阶段。

## 修复方案

1. 将 `perftest` 的 `mmap` 调用统一改为内核当前支持的 `prot/flags` 组合。
2. 保留失败时的结构化输出，但让校验器把这类结果升级为硬错误，而不是 warning。
3. 对旧数据重新解释：凡是由这类参数不匹配产生的 `MMAP_*` 结果，都不能写进正式结论。

## 修复后验证

修复后需要同时满足两层验证：

1. `perftest` 运行时，`MMAP_SPARSE` 和 `MMAP_FULL` 不再稳定地产生 `-1`。
2. `validate_logs.py` 对这类必需项失败会直接返回非零，阻断继续出图。

## 为什么这不是误判

如果只是单纯性能差，结果应该仍然是合法时间值，并能在多轮中重复出现。这里出现的是“测试根本没跑起来”的失败值，而且能直接由调用参数和内核检查逻辑解释，因此这不是统计波动，而是接口不匹配。

## 经验结论

1. 实验代码也是代码，接口协议不一致会直接毁掉结论。
2. 当结果出现 `-1`、`error` 这类哨兵值时，第一反应应是确认实验是否真正进入了有效路径。
3. 模块三必须把“功能失败”和“性能退化”严格区分开。

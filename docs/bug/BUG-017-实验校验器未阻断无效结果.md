# BUG-017: 实验校验器未阻断无效结果

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-017 |
| 分类 | 测试模块 (TEST) |
| 严重程度 | 高 |
| 初次记录时间 | 2026-03-22 |
| 复核日期 | 2026-03-25 |
| 影响文件 | `webui/validate_logs.py`, `webui/experiment_data.py`, `webui/plot_results.py` |

## 现象与触发条件

旧版模块三即便遇到明显无效的实验结果，也往往只是给 warning，然后继续画图、导出 JSON、生成报告。这样会产生一种非常危险的错觉：

- 脚本能出图；
- 文件都生成了；
- 看起来像“实验系统已经闭环”。

但实际上，底层数据可能已经失效。

## 影响范围

1. 无效结果可能继续流入论文图表。
2. 使用者容易把“脚本没报错”误认为“结果可信”。
3. 模块三失去作为质量门禁的意义，只剩一个自动出图器。

## 复现与观察

本问题在两类场景里最典型：

1. `MMAP_*` 等必需项返回 `-1` 或 `unit=error`；
2. 不同 `unit/batch/runs` 的历史数据被混在一起后，旧报告仍继续输出均值和结论。

这说明旧版校验器和聚合器没有真正扮演“阻断无效数据”的角色。

## 排查过程

1. 先从旧图表和日志中发现：明明底层存在失败值，报告仍然给出了看起来正常的比较结论。
2. 回看 `validate_logs.py`，发现它对一部分关键错误只给 warning，没有以非零状态退出。
3. 再看 `plot_results.py` 与 `experiment_data.py`，发现即使输入数据已经带有问题，后续链路仍会继续聚合和出图。
4. 最终确认问题不在单个测试场景，而在“质量门禁”缺失。

## 根因结论

根因是模块三早期更强调“尽量把产物生成出来”，而没有把“阻断无效数据”提升为一等目标。结果就是：

- 出图能力先有了；
- 数据质量门禁跟不上；
- 于是无效数据也能继续往前流。

## 修复方案

1. 将 testing 分支的关键实验项纳入必需项集合。
2. 对 `avg < 0`、`unit=error`、missing required tests、mixed-signature 等情况直接报错并返回非零退出码。
3. 让 `plot_results.py` 对 `invalid-dataset`、`insufficient-rounds` 做降级表述，而不是继续给出过强结论。
4. 保留 warning，但让 warning 回到“解释边界提醒”的位置，而不是替代硬错误。

## 修复后验证

修复后应能观察到：

1. 旧的无效日志集不再能通过校验器。
2. mixed-signature 数据不会再被静默求均值。
3. 报告中不再把无效数据包装成正式结论。

## 为什么这不是误判

这里的问题不是“校验器太严格”，而是旧版确实放过了本应阻断的失败数据。对论文实验来说，能不能及时拒绝坏数据，比能不能多生成一份图更重要。

## 经验结论

1. 自动化链路里，校验器必须是门禁，而不是提示器。
2. 对论文相关产物，宁可失败得明显，也不要“带病成功”。
3. 结构化实验系统的价值，不只是自动生成结果，更在于自动阻断不该相信的结果。

# BUG-020: `sysmon` 栈溢出导致监控卡住

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-020 |
| 分类 | 用户工具 (USER) |
| 严重程度 | 高 |
| 初次记录时间 | 2026-03-22 |
| 影响文件 | `user/sysmon.c` |

## 现象与触发条件

`sysmon` 本来是给模块四准备的监控导出工具，但早期版本一运行就卡在开头，典型表现是：

```text
$ sysmon 10 12
SYSMON_BEGIN interval=
```

随后程序停住、输出异常，或者根本没有后续 `SNAP/PROC` 记录。

## 影响范围

1. 模块四没有稳定的监控数据入口。
2. 容易让人误以为是 `getsnapshot()` 或内核监控接口有问题。
3. 如果不继续深挖，会把注意力错误地放到系统调用链，而不是用户态工具自身。

## 复现与观察

排查时最关键的观察点有两个：

1. 程序连最早期输出都打印不完整，说明问题可能发生在用户态准备阶段。
2. xv6 用户栈只有一页，而 `sysmon` 在 `main()` 里声明了较大的局部缓冲区。

## 排查过程

1. 先怀疑 `getsnapshot()`、`getptable()`、`getmemstat()` 其中之一返回异常，但这条线索解释不了“为什么连开头字符串都被破坏”。
2. 再检查 `sysmon` 的局部变量布局，发现 `struct pstat ps[64]` 加上其他结构体已经非常接近甚至超过 xv6 用户栈容量。
3. 结合输出停在 `SYSMON_BEGIN` 附近的症状，确认问题更像是栈被提前踩坏，而不是系统调用本身失败。
4. 把大缓冲区改到静态存储区后，程序恢复正常输出，进一步验证了这一判断。

## 根因结论

根因是用户态工具忽略了 xv6 的小栈约束：

- `sysmon` 在 `main()` 栈上分配了过大的结构体数组；
- 栈破坏发生在真正进入采样循环之前；
- 因此最早暴露出来的是输出异常和“看起来像卡住”。

## 修复方案

1. 将大缓冲区从局部变量迁移到静态全局区。
2. 在代码里补充注释，明确说明 xv6 用户栈很小，不适合在栈上堆大数组。
3. 保留 `sysmon` 的结构化输出格式，让修复不改变上层消费协议。

## 修复后验证

修复后应能稳定观察到：

- `SYSMON_BEGIN` 输出完整；
- 后续 `SNAP`、`PROC` 记录持续出现；
- 模块四和宿主机转换脚本可以正常消费这些数据。

## 为什么这不是误判

如果问题真在系统调用链，迁移缓冲区位置不应让症状完全消失。这里在不改内核接口的前提下，仅通过减少栈压力就恢复正常，说明根因确实在用户态栈使用方式。

## 经验结论

1. 在 xv6 里写用户态工具时，栈空间是必须优先考虑的资源约束。
2. “输出卡住”不一定是 I/O 问题，也可能是更早的栈破坏。
3. 给模块四准备的数据工具，稳定性优先级不低于功能完整性。

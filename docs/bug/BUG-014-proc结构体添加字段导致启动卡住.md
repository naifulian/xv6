# BUG-014: 启动卡住现象被早期误判为 proc 结构体扩容问题

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-014 |
| 分类 | 调度模块 (SCHED) |
| 严重程度 | 高 |
| 初次记录时间 | 2026-03-07 |
| 复核日期 | 2026-03-24 |
| 影响文件 | `kernel/sched/sched.h`, `kernel/sched/sched.c`, `kernel/sched/*.c`, `kernel/proc.h` |

## 问题描述

旧版 BUG-014 把“系统启动后无法进入 shell”直接归因为 `struct proc` 新增字段导致结构体变大。2026-03-24 在继续修复模块一审查项时，这个现象再次出现，并通过动态追踪重新定位后确认：

- `struct proc` 变大不是这类故障的充分根因。
- 当前仓库在保留 `heap_end`、`vmas` 等新字段的前提下，仍然可以稳定启动到 shell。
- 旧文档记录的是一次“回退改动后现象消失”的经验结论，但不是严格验证过的根因结论。

因此，本条 bug 需要从“结构体扩容导致启动卡住”修订为“启动卡住现象曾被误判为结构体扩容问题”。

## 复现现象

本轮复核时，启动阶段表象仍然是：

```text
xv6 kernel is booting
[随后长时间不进入 shell]
```

如果直接看表象，很容易再次把问题归到 `proc` 结构体变化上，因为启动链确实在进程调度和首个用户进程附近停住。

## 本轮重新定位过程

1. 先确认工作区能编译通过，但 `make CPUS=1 qemu` 无法稳定进入 shell。
2. 通过在 `userinit()`、默认调度器和 `forkret()` 上增加临时日志，确认系统实际上已经：
   - 选中了 `pid=1`；
   - 执行了 `fsinit(ROOTDEV)`；
   - 执行了 `kexec("/init", ...)`；
   - 返回过用户态并开始处理 `init` 的系统调用。
3. 继续用 `scheduler_test` 做动态验证时，发现切换到 `PRIORITY` 调度器会在内核态跳到异常地址，故障地址落在 `0x80009930` 一带，邻近的是 `syscalls` 表和调度器名称字符串，而不是合法函数入口。
4. 结合全量重编前后的行为差异，最终确认核心问题是：调度框架接口变更后，存在旧对象文件仍按旧版 `struct sched_ops` 布局解释函数指针，导致把数据地址当成代码跳转。

## 修正后的根因结论

### 直接根因

本轮真正确认的根因不是 `struct proc` 扩容，而是 **调度框架接口变更后的 ABI 不一致**：

- `kernel/sched/sched.h` 中 `struct sched_ops` 的字段布局发生变化；
- 如果没有强制全量重编，旧的调度器目标文件仍可能按旧偏移读取 `pick_next`、`proc_exit` 等函数指针；
- 结果是调度器切换或启动早期调度时，内核可能跳到字符串常量或数据表附近，表现为：
  - 开机后长时间不进入 shell；
  - `scheduler_test` 在策略切换时触发 `kerneltrap`；
  - 指令页错误地址落在非函数区间。

### 为什么旧结论会看起来“像是真的”

因为回退 `proc.h` 改动通常会连带触发部分对象重新编译，现象会暂时消失，所以当时容易误以为“结构体大小变化”就是根因。但这更像是 **回退动作顺手消除了 ABI 不一致**，而不是字段本身不可添加。

## 与本轮其他回归问题的关系

本次排查中还发现了一个独立问题：`exec` 路径里若把 `heap_end` 处理错，会导致 shell 启动后执行普通外部命令时在低地址 guard page 附近触发 `mappages: remap`。这个问题会放大“系统不稳定”的体感，但它不是 BUG-014 这条启动期误判的核心根因。

## 修复方案

1. 修改调度框架接口后，执行 `make -B -j2` 强制全量重编，避免残留旧 ABI 的调度器对象文件。
2. 继续保留 `struct proc` 新增字段，但要求相关初始化路径闭环：
   - `allocproc()` 初始化；
   - `freeproc()` 清理；
   - `fork()` / `exec()` 同步必要状态。
3. 把“能否启动到 shell”和“`scheduler_test` 是否通过”加入调度框架修改后的最小回归集。

## 修复后验证

本轮修复后已完成以下验证：

```text
make -B -j2                    -> 通过
make CPUS=1 qemu              -> 可稳定进入 shell
echo hi                       -> 通过
mmap_test                     -> 通过
scheduler_test                -> 通过
```

其中 `scheduler_test` 已能完成 `DEFAULT`、`FCFS`、`PRIORITY`、`SML`、`LOTTERY` 的切换和后续子进程测试，说明调度器对象表与当前接口布局已经重新对齐。

## 更新后的经验结论

1. **不要把“回退某个结构体改动后现象消失”直接写成根因**。
2. **修改带函数指针的核心接口时，要优先怀疑 ABI 不一致和增量构建残留**。
3. **启动验证不能只看“能否编译”，至少要跑到 shell，并补一个策略切换测试**。
4. **BUG 文档要区分现象、触发条件和根因，避免后续排障被旧结论带偏**。

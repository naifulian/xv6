# BUG-026: 模块四前端命令示例反引号导致页面停在初始化

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-026 |
| 分类 | USER |
| 严重程度 | 高 |
| 初次记录时间 | 2026-03-26 |
| 影响文件 | `dashboard/js/views/home.js`, `dashboard/js/views/processes.js`, `dashboard/js/views/scheduler.js`, `dashboard/js/views/memory.js`, `dashboard/js/views/events.js` |

## 现象与触发条件

在 2026 年 3 月 26 日的模块四运行链路里，`dashboard/scripts/start_monitor.sh` 已经能够正常启动 bridge，`dashboard/runtime/dashboard-telemetry.log` 也持续写入 `SNAP/PROC` 快照，但浏览器里的模块四页面仍长期停留在首页初始占位文本“初始化中”。

触发条件很简单：只要浏览器加载新的模块四首页，就会在解析前端模块时出错，因此 UI 根本来不及把初始占位文本替换成真实状态。

## 影响范围

这个问题影响的是模块四前端入口的可用性，而不是 telemetry 链路本身：

- bridge 接口其实已经在线，`/api/meta` 和 `/api/dashboard` 都能返回正常 JSON。
- guest 侧 `dashboardd` 也已经稳定产出遥测。
- 但由于前端模块在解析阶段报错，页面会误导用户继续怀疑 bridge 或 telemetry 没通。

因此，这属于典型的“展示层卡死伪装成数据链路未通”的误判风险。

## 复现与观察

复现时可以观察到以下事实：

1. `dashboard/runtime/dashboard-telemetry.log` 中已经存在大量 `DASHBOARDD_BEGIN`、`SNAP` 和 `PROC` 记录。
2. `curl -s http://127.0.0.1:8000/api/meta` 能返回有效 `generated_at`、`source` 与 `parse_error=null`。
3. 页面却仍停在 `dashboard/index.html` 里的初始文案“初始化中”。

进一步用本地 Node mock 导入前端入口时，能够稳定复现语法错误：

- 修复前：`SyntaxError: Unexpected identifier 'echo'`
- 修复后：`import-ok`

这个差异说明问题不在网络请求，而在前端模块本身的解析阶段。

## 排查过程

1. 先检查启动脚本 `dashboard/scripts/start_monitor.sh`，确认 bridge 默认读取 `dashboard/runtime/dashboard-telemetry.log`。
2. 再检查 `dashboard/bridge/app.py`、`dashboard/bridge/state.py` 和实际运行时接口，确认 `/api/meta`、`/api/dashboard` 都能正常返回数据。
3. 因为 telemetry 和 API 都正常，于是把排查重点转到前端入口 `dashboard/js/main.js` 及其依赖模块。
4. 对 `main.js` 做最小浏览器环境 mock 导入时，立即得到 `SyntaxError: Unexpected identifier 'echo'`。
5. 继续检查各视图文件，发现多个页面把命令示例写成了 Markdown 风格反引号，例如 `` `echo hello` ``、`` `ps` ``、`` `mmap_test` ``。这些文本本身位于 JavaScript 模板字符串里，导致模块在解析阶段被未转义反引号提前截断。

## 根因结论

根因不是 bridge 离线，也不是 telemetry 没产出，而是模块四前端在多个视图模板字符串中直接嵌入了 Markdown 风格反引号命令示例。

由于这些反引号和外层 JavaScript 模板字符串共用同一种定界符，浏览器在解析模块时会提前结束字符串，后续文本再被当成 JavaScript 标识符解析，于是报出 `Unexpected identifier 'echo'` 这类语法错误。

这说明模块四此前虽然完成了分层刷新和多页面拆分，但命令示例的展示方式没有同步从 Markdown 语法切换到 HTML 语义，留下了一个会直接阻断前端启动的低层错误。

## 修复方案

修复方式保持最小：

1. 将五个视图文件中的 Markdown 反引号命令示例改成 HTML `<code>...</code>`。
2. 保留原有文案和展示意图，只修正会破坏 JavaScript 模板字符串解析的写法。
3. 修复文件包括：
   - `dashboard/js/views/home.js`
   - `dashboard/js/views/processes.js`
   - `dashboard/js/views/scheduler.js`
   - `dashboard/js/views/memory.js`
   - `dashboard/js/views/events.js`

## 修复后验证

修复后已完成以下验证：

1. 再次搜索模块四视图目录，不再存在这组会破坏模板字符串的命令反引号写法。
2. 用本地 Node mock 再次导入 `dashboard/js/main.js`，结果变为 `import-ok`，说明前端入口已能完整解析并执行。
3. bridge 侧仍能返回正常 API，说明本次修复没有改变现有数据链路，只修正了前端启动阶段的阻断点。

## 经验结论

1. 模板字符串里的展示文案不是“纯文案”，它仍然要服从 JavaScript 语法边界。
2. 在 HTML 模板中展示命令示例时，应优先使用 `<code>`，不要直接套用 Markdown 风格反引号。
3. 当页面停在静态初始占位状态时，不能只怀疑后端或数据链路，还要优先确认前端入口脚本是否真的成功执行。

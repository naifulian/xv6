# BUG-018: 用户态监控工具未覆盖新调度器

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-018 |
| 分类 | 用户工具 (USER) |
| 严重程度 | 中 |
| 初次记录时间 | 2026-03-22 |
| 影响文件 | `user/ps.c`, `user/sysmon.c`, `Makefile` |

## 现象与触发条件

模块一已经扩展到 9 种调度器，但用户态工具仍停留在旧版本世界观：

1. `ps` 只能正确显示早期调度器名称。
2. 模块四需要的结构化快照数据没有稳定出口。

这导致一个尴尬局面：内核功能已经有了，但用户态观测层仍像停在旧项目版本。

## 影响范围

1. 新调度器的状态展示不完整。
2. 模块四缺少稳定的数据入口。
3. 调试时无法方便地把 `getsnapshot/getptable/getmemstat` 串成结构化输出。

## 复现与观察

排查时可以观察到：

- `ps` 对新增调度器名称显示异常或不完整；
- 没有一个统一的用户态工具能按固定格式导出系统快照；
- Web 展示层如果想做，必须自己重新拼一套数据入口。

## 排查过程

1. 先确认内核已经支持更多调度器，并且相关系统调用接口存在。
2. 再查看用户态工具，发现名称映射和展示逻辑没有同步扩展。
3. 同时评估模块四需求，确认仅靠现有 `ps` 输出不适合做稳定数据源。
4. 因此问题被重新定义为：不仅要补名字映射，还要补一个面向监控的结构化导出工具。

## 根因结论

根因不是单一显示 bug，而是“用户态观测层没有随着内核能力一起演进”。这类问题在功能快速扩展时很常见：核心实现先走，观测和展示层滞后。

## 修复方案

1. 扩展 `ps` 的调度器名称映射，覆盖当前全部策略。
2. 新增 `sysmon`，统一输出 `SNAP/PROC` 等结构化记录。
3. 将 `_sysmon` 纳入 `fs.img`，让它成为默认可运行工具，而不是只存在于源码里。

## 修复后验证

修复后应能做到：

- `ps` 正确显示新增调度器名称；
- `sysmon` 能稳定输出结构化快照；
- 模块四和宿主机侧脚本可以直接消费这些输出，而不必再猜格式。

## 为什么这不是误判

这个问题不是“显示层小瑕疵”。对模块四而言，如果没有稳定的数据入口，Web 页面就只能演示静态图，无法展示系统状态。因此它本质上是模块四入口缺失问题的一部分。

## 经验结论

1. 每扩展一类内核能力，都应同时检查用户态观测工具是否需要跟进。
2. 对展示层最友好的不是人类可读输出，而是结构化、稳定、可解析的输出。
3. 模块四的很多问题，其实要在模块一/模块三的接口层提前解决。

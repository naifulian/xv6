# Bug 档案索引

`docs/bug/` 是本项目的问题证据库。它的价值不只是说明“修了什么”，更重要的是保留“为什么会误判、如何排查、为什么当前修法可信”的完整链路。对于本项目的论文写作而言，这些 bug 文档既是工程真实性的来源，也是系统设计如何逐步收敛的重要证据。

## 阅读定位

建议把这个目录理解为两层材料：

1. 对主文档的补充证据库
2. 对系统演进过程的复盘档案

总架构文档和模块文档只吸收关键 bug 的精简闭环，例如“问题暴露了什么设计缺口、修复后架构如何变化、为什么当前验证足以说明修复可信”；而 `docs/bug/` 则保留完整的现象、排查、根因、修复和验证过程。

## 典型排查模式

### 1. 先分清“功能错了”还是“观测错了”

很多 bug 一开始看像内核逻辑问题，最后发现是测试、监控或实验口径错了。BUG-016、BUG-017、BUG-018、BUG-019 都属于这一类。

### 2. 不把“回退后现象消失”直接当根因

BUG-014 是典型例子。回退某个改动后现象消失，可能只是顺手触发了全量重编或改变了时序，不代表那个改动本身就是根因。

### 3. 在 xv6 里优先怀疑资源约束

xv6 的用户栈、页表语义、QEMU 单核 tick 分辨率都很小。BUG-020 和模块三多条方法学 bug 都直接受这些约束影响。

### 4. 对论文相关 bug，必须补“为什么这不是误判”

只写修复方案不够。像 BUG-016、BUG-017、BUG-019 这类影响实验结论的 bug，必须同时保留：

- 旧结果为什么不可信
- 新护栏如何阻断同类错误
- 修复后如何验证口径已经收敛

## 编写规范

新的 bug 文档统一尽量包含以下部分：

1. 基本信息
2. 现象与触发条件
3. 影响范围
4. 复现与观察
5. 排查过程
6. 根因结论
7. 修复方案
8. 修复后验证
9. 经验结论

不是每条都必须一样长，但至少要回答清楚：发生了什么、为什么这不是误判、为什么当前修法可信。

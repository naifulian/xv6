
# BUG-025: 模块四分层刷新前仍是慢速整页轮询

## 基本信息

| 项目 | 内容 |
|------|------|
| 编号 | BUG-025 |
| 分类 | 用户工具 (USER) |
| 严重程度 | 中 |
| 初次记录时间 | 2026-03-26 |
| 影响文件 | `dashboard/js/app.js`, `dashboard/index.html`, `dashboard/css/style.css`, `user/init.c` |

## 问题描述

在模块四第一次收口前，前端仍然使用统一的整页轮询刷新：总览、进程、事件、调度和内存都按同一个节奏更新。

这会带来两个明显问题：

- 页面感觉慢，短命事件很容易被覆盖掉。
- 监控体验更像“慢速看板”，不像任务管理器那种分层观察界面。

## 复现现象

修复前的典型体验是：

- 总览信息更新滞后。
- 进程表和事件流会互相抢刷新节奏。
- 短时间内发生的状态变化很难被稳定看到。

## 排查过程

1. 先看前端脚本，发现刷新逻辑被压在统一的 `REFRESH_MS` 里。
2. 再看首页结构，发现说明卡和监控卡混在一起，页面语义也不够清晰。
3. 对照审查文档中的问题三，确认“刷新太慢”不是观感问题，而是展示层节奏设计问题。
4. 在提交 `86acf06` 后，前端改成了总览、进程详情、事件流分层刷新。

## 根因结论

根因是前端一开始把所有数据都当成同一类“看板更新”来处理，没有按照页面关注点拆刷新节奏。

这会让不同粒度的状态变化被同一个刷新频率覆盖，从而降低 live monitor 的可读性。

## 修复方案

1. 总览层、进程主表/详情层、事件层拆成不同刷新频率。
2. 首页补上分层刷新说明，让页面语义更像任务管理器而不是一个慢速状态看板。
3. 将短命事件的展示单独收口，避免被整页刷新节奏淹没。

## 修复后验证

修复后已确认：

- 总览、进程和事件各自按不同节奏更新。
- 页面观感更接近任务管理器。
- 短命状态更容易被观测到。

## 经验结论

1. 监控页不是“越统一越好”，而是要按信息粒度拆节奏。
2. 页面的刷新策略本身就是产品体验的一部分。
3. 短事件看不到，很多时候不是数据没来，而是刷新设计太粗。

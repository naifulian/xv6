# 系统架构与设计闭环

## 项目背景与研究目标

本项目以 MIT xv6-riscv 为基础，对其进行面向本科毕业设计的系统化增强。原始 xv6 的教学价值在于结构清晰、代码规模可控，但其默认实现更强调“可讲解的最小内核”，而不是“可支撑完整系统研究流程的平台”。如果直接以原始 xv6 为对象开展调度算法比较、内存管理机制分析和性能实验，往往会遇到三个根本问题：第一，内核机制本身过于单一，缺少可比较对象；第二，测试与实验链路不完整，难以证明结论可信；第三，观测与展示接口薄弱，无法把实现、验证与结果组织成规范的论文闭环。

因此，本项目的总体目标不是把 xv6 改造成工业级操作系统，而是在保留其教学系统特征的前提下，构建一个“可研究、可验证、可实验、可展示”的增强型平台。更具体地说，项目希望完成以下四项任务：

1. 在内核层扩展调度与内存管理机制，使 xv6 具备可比较、可分析的实现基础。
2. 在测试层建立统一回归框架，使关键机制具备可重复验证能力。
3. 在实验层形成结构化数据采集、校验与分析链路，使性能结论具备可追溯性。
4. 在展示层复用已有结构化产物，为答辩和后续维护提供更直观的观察入口。

这四项任务共同构成了本项目的毕设闭环：内核增强提供研究对象，测试框架提供正确性约束，实验链路提供定量证据，展示层提供结果消费入口。

## 原始 xv6 的局限与扩展必要性

### 调度能力局限

原始 xv6 仅提供单一的轮转调度（Round Robin）策略。对于“调度算法优化”这一毕设主题而言，这种单策略结构难以支撑对比分析，也不具备“算法切换、行为观测、结果验证”的基本能力。如果在此基础上直接堆叠多个算法实现，而缺少统一调度接口，则代码很容易退化为 `if(policy == ...)` 风格的分支集合，既不利于维护，也不利于论文叙事。

### 内存管理局限

原始 xv6 的物理页分配和地址空间管理偏向最小可用实现，缺少 Buddy、Copy-on-Write、Lazy Allocation、`mmap` 等现代教学中常见的机制。更重要的是，原始实现没有为这些机制预留统一的地址空间元数据与缺页处理框架。如果直接局部打补丁，很容易出现每个 feature 自成一摊、彼此不共享语义的情况，最终难以形成完整的系统设计章节。

### 测试能力局限

仅凭“程序能跑”无法证明内核增强实现的正确性。对于调度器、懒分配、COW、`mmap` 这类涉及时序、权限、缺页和地址空间边界的机制，若没有专门的测试入口和明确断言，许多错误只能在后续实验甚至答辩阶段才暴露出来，这会严重影响结论可信度。

### 实验方法局限

原始 xv6 并不提供面向系统实验的数据采集链路。即使实现了多个机制，如果实验仍停留在“手动运行一次、复制控制台输出、人工画图”的方式，就无法保证 provenance、重复性与数据一致性，也很难支撑正式论文写作。

### 可观测与展示能力局限

教学操作系统的控制台输出通常适合调试，但不适合稳定消费。若展示层直接依赖原始控制台日志，则上层系统会与底层输出格式强耦合，既不稳定，也不利于后续扩展。

## 总体架构设计

为解决上述问题，项目采用“四模块协同”的总体架构：

```text
模块一：内核机制增强
  -> 提供多调度策略、现代内存机制与统一观测接口

模块二：测试与回归框架
  -> 对模块一关键行为建立强断言与 smoke 保护层

模块三：实验采集、校验与分析链路
  -> 将模块一能力转化为结构化实验结果与论文图表

模块四：展示与监控层
  -> 复用模块三产物与系统快照，为结果展示提供入口
```

四个模块不是并列堆放的功能点，而是自底向上逐层承接：

1. 模块一解决“系统能做什么”的问题。
2. 模块二解决“这些能力是否实现正确”的问题。
3. 模块三解决“这些能力是否能形成可信结论”的问题。
4. 模块四解决“这些结论如何被消费和展示”的问题。

从工程角度看，这一架构避免了“先写完所有功能，再临时补实验和文档”的松散流程；从论文角度看，这一架构使系统设计、实验方法和结果展示之间存在清晰的依赖关系。

## 四模块协同闭环

### 模块一：内核机制增强层

模块一由两条主线组成：调度器扩展与内存管理扩展。

调度部分通过 `kernel/sched/sched.h` 中的 `sched_ops` 接口，把 9 种策略统一纳入同一调度框架。核心钩子包括 `proc_init`、`pick_next`、`proc_exit`、`proc_tick` 和 `proc_wakeup`。这样一来，算法差异被限制在策略实现内部，而生命周期接线由统一框架负责。其意义在于：论文可以把“调度框架设计”作为一项独立成果来写，而不是只列出算法清单。

内存部分则围绕 `kernel/vm.c` 的统一缺页路径展开。Buddy 分配器负责物理页层面的组织，`proc.h/proc.c` 引入 `heap_end + vmas[] + sz` 作为地址空间元数据，`sysproc.c` 中的 `sys_mmap/sys_munmap` 负责用户态接口落地，`vmfault()` 将 Lazy Allocation、匿名私有 `mmap` 和部分地址合法性判断集中到同一路径中。这种设计避免了“每种机制单独处理页错误”的分裂结构。

### 模块二：正确性保护层

模块二的核心入口并不在 `user/` 目录，而是通过 `Makefile` 将 `tests/src/test_runner.c` 与各类测试对象文件链接为用户态程序 `_test_runner`。测试分为 Buddy、`mmap`、COW、Lazy、Integration、Boundary、Scheduler 七类，共 68 个用例。

该层的价值不在于追求形式化证明，而在于把最容易被误判的内核行为变成可重复回归的断言。例如：

1. `wait()` 返回值按 xv6 原生语义解释，而不是套用 POSIX 状态字宏。
2. 调度测试区分“强断言场景”和“仅能做 smoke 的场景”。
3. Lazy/COW 等路径不仅验证“是否能运行”，还验证缩容、共享、权限与缺页等关键语义。

模块二使模块一不再只是“实现功能”，而是“实现并持续校验功能”。

### 模块三：实验方法与证据层

模块三围绕 `user/perftest.c -> experiments/collect/collect_data.sh -> experiments/analyze/experiment_data.py -> experiments/analyze/validate_logs.py -> experiments/analyze/plot_results.py` 形成完整数据链路。

该链路解决了三个长期困扰教学系统实验的问题：

1. 如何稳定采集：`collect_data.sh` 使用结构化 run 目录、`manifest.tsv`、raw/clean log、run status 区分 `complete/partial/empty`。
2. 如何安全聚合：`experiment_data.py` 保留 round-level、sample-level 和 per-run 明细，并在 mixed-signature 情况下拒绝聚合。
3. 如何阻断坏数据：`validate_logs.py` 将 missing required tests、invalid result、mixed-signature 等问题作为门禁错误，而不是普通 warning。

因此，模块三不只是“画图脚本集合”，而是将实验过程本身规范化，使最终结果具备论文级 traceability。

### 模块四：展示与消费层

模块四当前目录已经统一为 `dashboard/`。系统运行期状态通过 `user/sysmon.c` 输出结构化 `SNAP/PROC` 记录，再由 `dashboard/parse_sysmon.py` 转换为 `dashboard/data/snapshots.json`，供前端页面消费；模块三正式实验产物则位于 `experiments/outputs/`。

这种做法的意义在于：展示层只承担消费职责，不再反向定义底层协议，从而保证数据语义的一致性与可维护性。

## bug 驱动的设计收敛

本项目的架构并非一开始就完全成型，而是在多轮 bug 排查中不断收敛。与论文写作相关的关键结论，不应只写“最后方案是什么”，还应说明“为什么原方案不够、为什么当前修法可信”。本项目的多个关键 bug 正是推动架构从点状 feature 走向闭环系统的转折点。

### 调度框架侧的收敛

`BUG-007` 暴露了调度器切换缺少锁保护的问题。该问题表明，调度策略并不是局部配置，而是影响全局共享状态的核心控制平面。因此，后续设计明确了策略切换必须经过统一框架，并通过共享状态与生命周期钩子一并管理，而不能把调度算法视为可随意插拔的局部逻辑。

`BUG-014` 则进一步说明，对核心调度接口的修改不仅是“源码改动”，也是接口契约变更。启动卡住现象最初被误判为 `struct proc` 扩容问题，复核后才确认更直接的风险来自调度框架接口布局变化与对象文件 ABI 不一致。这个案例使得项目在工程上形成了更严格的回归要求：涉及调度框架接口时，必须至少验证编译、进入 shell 和调度器切换测试。

### 内存管理侧的收敛

`BUG-012` 表明，仅在用户写缺页时处理 COW 还不够，内核向用户地址空间写回数据的 `copyout()` 路径同样必须理解 COW 语义，否则“用户态看起来正确”的设计在系统调用写回时仍会破坏页共享与权限逻辑。这一问题推动了项目从“多个 feature 各写一段补丁”转向“统一页表与缺页语义”的设计思路。

### 实验方法侧的收敛

`BUG-016`、`BUG-017` 和 `BUG-019` 共同推动了模块三从“能出图”走向“能守住结论边界”。`BUG-016` 暴露出工作负载与内核接口不匹配的问题；`BUG-017` 暴露出校验器缺少门禁能力的问题；`BUG-019` 则指出实验场景区分度不足，即使结果看起来稳定，也可能无法支撑有效比较。三者共同说明：性能实验的可信度并不只取决于脚本是否运行成功，而取决于工作负载设计、数据质量门禁和统计口径是否一致。

### 展示链路侧的收敛

`BUG-018` 和 `BUG-020` 说明展示层也必须遵守底层系统约束。前者暴露用户态监控工具未能覆盖新增调度器，后者暴露 xv6 用户栈容量极小，用户态监控程序如果在栈上放置大缓冲区，会在真正进入采样前就破坏自身状态。这些问题使得模块四最终采用“轻协议、静态产物复用、监控快照单独转换”的稳妥路线。

## 文档与证据的对应关系

为了让论文叙事与工程证据保持一致，本项目将文档划分为三个层次：

1. 本文档负责讲清总体架构、模块关系、设计闭环与论文写法边界。
2. `docs/module1-4/` 负责讲透各模块的原理、接口、流程、取舍与验证。
3. `docs/bug/` 负责保留完整的现象、排查、根因、修复与验证证据。

这种组织方式的目的，是避免把所有历史细节塞入主文档，也避免主文档只剩空泛概述。论文正文可以主要依赖本文档和模块文档，答辩时如需进一步说明具体 bug 如何定位与修复，再回到 `docs/bug/` 查证。

## 论文写法边界

### 可以明确写入论文的内容

1. 本项目已经在 xv6 上实现多调度策略与多种内存管理机制，并以统一框架组织，而非独立功能点堆叠。
2. 本项目建立了从功能实现、测试回归、实验采集到展示消费的完整闭环。
3. 正式实验数据已经形成结构化口径，可支撑方向性结论、反例分析和区间化解释。

### 需要保留边界的内容

1. `SJF/SRTF` 仍是教学化估计模型，而非工业级 burst 预测系统。
2. `MLFQ` 当前适合写成简化版多级反馈队列，不宜表述为完整 Linux 风格实现。
3. `mmap` 当前仅支持匿名私有懒映射原型，不支持文件映射、共享映射与回写。
4. 模块四当前更适合写成“方案明确、链路打通、展示层待进一步工程化”，不宜写成完整 Web 平台已全部完成。

### 实验结论的表述原则

1. 正式结论应基于 `experiments/logs/thesis_final` 及其派生产物。
2. 对高波动指标，应强调 95% CI、CV 和 warning 所表达的是解释边界，而不是简单忽略。
3. 调度与内存实验结论都应写成“在当前 xv6/QEMU 单核教学环境和既定 workload 下观察到的趋势”，而不是面向所有操作系统的普遍定律。
